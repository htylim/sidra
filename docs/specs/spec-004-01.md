# Capture Readable Page Context Explicitly Implementation Plan

## Overview

Implement issue 004: `Capture + Send` must capture readable context from the active browser tab only when the user explicitly sends with capture, attach that context to the outgoing prompt as untrusted reference material, and show compact markers in the transcript instead of raw page content.

## Current State Analysis

Sidra already has the main ownership boundaries that issue 004 needs. The extension owns active page state, URL sessions, draft prompt state, and bridge-facing commands. The extension bridge boundary owns Native Messaging connection state and session-start sequencing. The bridge owns provider sessions and receives `pageContext` in its provider input type.

What is missing is the capture boundary itself. The current `Capture + Send` button is only a label; it calls the same plain prompt path used by `sendPrompt`, so no active tab is captured, no page context is built, and no context marker is inserted.

The issue file still says issue 003 is open, but the current codebase already contains URL-scoped session behavior. This plan treats the current code as the source of truth and builds on the existing `UrlSessionStore`, `PageIdentity`, and `BridgeSessionCoordinator` modules.

## Desired End State

At the end of this plan:

- `Capture + Send` is a separate application command from plain `sendPrompt`.
- Capture happens only inside that command, at click/Enter time, and never during side panel open, tab switch, active page tracking, or background activity.
- Capture reads the currently active tab, extracts a cloned document snapshot, recomputes the URL session key with captured canonical URL metadata, and sends the prompt through the correct URL session.
- Readability extraction uses `@mozilla/readability` against a cloned document. If Readability is missing or too short, Sidra falls back to `document.body.innerText`. If there is no usable text, Sidra sends metadata only.
- `session.send` carries structured `pageContext`; the provider-facing prompt formatter wraps captured content as untrusted reference material before a real provider consumes it.
- The transcript shows `Page context attached` or `Page metadata attached` before the user prompt. It never dumps captured page text.
- The current page card reflects the latest context state for the active URL session.

### Key Discoveries

- Architecture assigns capture orchestration to `apps/extension` and specifically names `CaptureService` as the owner of active-tab capture, extraction, size decisions, and page-context construction: `docs/ARCHITECTURE.md:9`, `docs/ARCHITECTURE.md:86`, `docs/ARCHITECTURE.md:92`.
- `ActivePageTracker` intentionally reads only tab URL/title metadata and must not use scripting or content extraction: `docs/ARCHITECTURE.md:89`, `apps/extension/src/active-page.ts:51`.
- URL session ownership already lives in `UrlSessionStore`; active sessions carry `contextState`, transcript, draft prompt, running state, and `clientSessionId`: `apps/extension/src/url-session-store.ts:5`, `apps/extension/src/url-session-store.ts:7`, `apps/extension/src/url-session-store.ts:81`.
- `resolvePageIdentity` already prefers canonical URL, strips hashes/tracking parameters, and rejects unsupported schemes: `apps/extension/src/page-key.ts:42`, `apps/extension/src/page-key.ts:68`.
- `SidePanelController` currently exposes only plain `sendPrompt`; the view's `Capture + Send` button calls that plain command: `apps/extension/src/side-panel-controller.ts:30`, `apps/extension/src/side-panel-controller.ts:139`, `apps/extension/src/side-panel-view.tsx:15`, `apps/extension/src/side-panel-view.tsx:90`.
- `BridgeSessionCoordinator` currently queues prompt strings and posts `session.send` without `pageContext`: `apps/extension/src/bridge/session-coordinator.ts:45`, `apps/extension/src/bridge/session-coordinator.ts:233`.
- The protocol already has a minimal `PageContext`, but it does not model readable vs metadata-only context or optional metadata fields from the PRD: `packages/protocol/src/index.ts:5`.
- The bridge provider interface already accepts `pageContext` and forwards it from validated protocol messages to the provider session: `apps/bridge/src/session-manager.ts:3`, `apps/bridge/src/index.ts:30`.
- Product requirements specify explicit capture only, active-tab recomputation at send time, Readability fallback, metadata-only fallback, untrusted wrapping, and compact transcript markers: `docs/prd-v1.md:235`, `docs/prd-v1.md:248`, `docs/prd-v1.md:252`, `docs/prd-v1.md:265`, `docs/prd-v1.md:301`, `docs/prd-v1.md:324`.

## What We're NOT Doing

- No settings-backed readable content size limit. Issue 005 owns that.
- No bridge hard payload ceiling. Issue 005 owns that.
- No full DOM capture mode, prompt options popover behavior, or DOM size handling. Issue 006 owns that.
- No split `Capture + Send` / plain `Send` UI, send-mode selector, or post-capture default switch to plain `Send`. `docs/issues/issue-004-bis.md` owns that follow-up.
- No quick action configuration changes. Issue 007 owns that.
- No streamed Markdown rendering, cancellation UI, permissions, heartbeats, Codex provider integration, or Native Messaging installer work.
- No persistent history, no pre-send preview, no passive capture, no recurring capture, and no content logging.
- No support for unsupported URL schemes beyond surfacing capture unavailable state.

## Implementation Approach

Keep the feature behind deep boundaries:

- `CaptureService` owns active-tab capture and extraction. It is the only extension module that calls `chrome.scripting.executeScript`.
- `ActivePageTracker` remains a metadata-only passive tracker.
- `SidePanelController` owns the `captureAndSend` application command and composes capture results with URL session state.
- `UrlSessionStore` owns session-scoped context state and delegates prompt submissions to the coordinator.
- `BridgeSessionCoordinator` owns protocol ordering and should support prompt submissions that include optional `PageContext` plus a transcript marker.
- `packages/protocol` owns the structured `PageContext` contract and runtime validation.
- The bridge owns provider-facing prompt formatting. The extension should send structured context, not duplicate captured text into the user prompt string. Provider adapters must receive a finalized provider prompt, not raw `pageContext`, so a future provider cannot accidentally bypass untrusted-context wrapping.

## Testing Strategy

This section complements the phase-level verification. It does not replace the red/green checks inside each phase.

### Unit Tests

- Protocol parser accepts and rejects each `PageContext` variant.
- Capture service covers explicit invocation, metadata extraction, canonical URL recomputation, Readability success, fallback to body text, metadata-only fallback, capture unavailable, and whitespace normalization.
- Session coordinator covers prompt ordering, queued submissions, context marker ordering, context payload posting, bridge disconnect cleanup, and plain send behavior.
- URL session store covers session-scoped context state, context state reset on New Chat, and inactive session isolation.
- Controller covers no passive capture, bridge-blocked capture prevention, active-tab-at-click routing, canonical session key recomputation, metadata-only behavior, and capture unavailable behavior.
- View covers button/Enter wiring, disabled unsupported state, marker rendering, and no raw content rendering.

### Integration Tests

- Existing bridge tests continue to prove provider sessions are scoped by `clientSessionId`.
- Bridge prompt formatter tests prove the provider-facing prompt is safe before the Codex provider exists.
- Manual smoke verifies the side panel page and explicit capture UI behavior with the local mock bridge.

### Manual Testing Steps

1. Open a supported article page and Sidra side panel.
2. Do not send anything; verify no capture marker appears.
3. Type a prompt and press `Capture + Send`.
4. Verify `Page context attached` appears before the user prompt.
5. Verify raw page text is not displayed in the transcript.
6. Test a short/no-text page and verify `Page metadata attached`.
7. Test `chrome://extensions` or another unsupported page and verify capture is unavailable and no prompt is sent.
8. Switch tabs after opening Sidra, then press `Capture + Send`; verify the active tab at click time is the captured session.

## Performance Considerations

- Capturing full HTML before Readability can be expensive on very large pages. Issue 005 will add size limits. For this issue, do not silently truncate.
- Keep extraction synchronous after the document snapshot is received, but isolate it in `CaptureService` so size-limit policy can be added without touching React.
- Do not store captured raw text in transcript state. Only send it through protocol and keep marker metadata in session state.
- Avoid passive capture entirely; this is both a privacy requirement and a performance guard.

## Migration Notes

- No persisted data migration is needed because URL sessions are in-memory.
- `PageContext` protocol validation changes are source-compatible only inside this repo because no external bridge clients are supported yet.
- Adding `@mozilla/readability` changes the extension dependency graph and lockfile.
