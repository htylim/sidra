# URL-Scoped Chat Sessions Implementation Plan

## Overview

Implement issue 003: visible chat state follows the active browser page, and each Page Key has its own URL Session while the side panel is alive.

This is extension-owned application behavior. The Native Messaging Bridge already accepts opaque `clientSessionId` values and already maps them to provider sessions, so this plan keeps bridge protocol shape unchanged and builds the missing browser-side URL Session ownership.

## Current State Analysis

Sidra currently has the bridge/session boundary from issues 001 and 002, but the side panel still has only one browser-side session.

### Key Discoveries

- `apps/extension/src/side-panel-controller.ts:43` creates one `BridgeSessionCoordinator` at controller construction time, using one `createClientSessionId()` result at `apps/extension/src/side-panel-controller.ts:45`.
- `apps/extension/src/side-panel-controller.ts:14` exposes one `activeSession` snapshot. There is no Page Key, active tab identity, per-page draft, or per-page context state in the controller snapshot.
- `apps/extension/src/side-panel-view.tsx:10` stores the draft prompt in local React state. That makes draft restoration across URL Sessions impossible without remounting or lifting ownership.
- `apps/extension/src/side-panel-view.tsx:34` renders a fixed Current Page Card with `Current page` and `No context sent yet`; it does not reflect active page identity or session-specific context state.
- `apps/extension/src/bridge/session-coordinator.ts:40` owns one extension-side provider session view: start sequencing, pending prompts, transcript updates, and bridge event filtering by `clientSessionId`.
- `apps/extension/src/bridge/session-coordinator.ts:188` ignores `session.started` messages for other `clientSessionId` values, and `apps/extension/src/bridge/session-coordinator.ts:202` ignores `agent.event` messages for other sessions. Multiple coordinators can safely share the same transport.
- `apps/bridge/src/session-manager.ts:37` already owns bridge-side provider sessions by `clientSessionId`. `apps/bridge/src/session-manager.ts:45` stores sessions in a map, and `apps/bridge/src/session-manager.test.ts:6` already proves distinct client IDs create distinct provider sessions.
- `packages/protocol/src/index.ts:13` already requires `clientSessionId` on session protocol commands and already supports optional `pageContext` on `session.send`; issue 003 does not need a protocol change.
- `apps/extension/public/manifest.json:6` already declares the `tabs` permission needed to read active tab URL/title metadata.
- `docs/prd-v1.md:200` defines URL-scoped session behavior: prefer canonical URL, otherwise current URL without hash, strip common tracking parameters, preserve non-tracking query parameters, and keep sessions only in memory.
- `UBIQUITOUS_LANGUAGE.md:19` defines URL Session, `UBIQUITOUS_LANGUAGE.md:20` defines Page Key, and `UBIQUITOUS_LANGUAGE.md:21` defines Client Session ID. Use these terms in code and docs.

## Desired End State

After this plan is implemented:

- The side panel tracks the active browser tab/page through a named Chrome boundary.
- A pure Page Key normalizer chooses canonical URL when provided, otherwise current URL, removes fragments, strips common tracking parameters, and preserves meaningful query parameters.
- The extension owns a `UrlSessionStore` that maps Page Keys to URL Sessions.
- Each URL Session owns:
  - Page Key and display page identity.
  - One Client Session ID.
  - One `BridgeSessionCoordinator`.
  - One Transcript through that coordinator.
  - One Draft Prompt.
  - One visible running/start/pending state derived from that coordinator.
  - One Context State, defaulting to `No context sent yet`.
- Switching active tabs or URLs switches the visible URL Session without leaking another page's Transcript or Draft Prompt.
- Returning to a previously seen Page Key restores its in-memory Transcript and Draft Prompt.
- Sending from different Page Keys uses different Client Session IDs. Returning to a Page Key reuses its existing Client Session ID and provider session state.
- The React view is controlled by the controller snapshot for Draft Prompt and Current Page Card state.
- Page identity and URL Session selection work even when the Native Messaging Bridge is unavailable, checking, or in an error state. Bridge readiness only gates sending prompts.
- No page content is captured passively. Issue 003 may read tab URL/title metadata through `chrome.tabs`, but it must not inject scripts or read DOM content.

## Implementation Approach

Build the missing extension ownership as deep modules:

1. `PageKey` normalizer: pure code, easy to test, no Chrome dependency.
2. `ActivePageTracker`: Chrome tab metadata boundary. It reads active tab URL/title and subscribes to tab/window changes. It must not use `chrome.scripting`.
3. `UrlSessionStore`: Page Key to URL Session map. It creates one `BridgeSessionCoordinator` per Page Key and owns Draft Prompt plus Context State.
4. `SidePanelController`: composition root for `BridgeConnection`, `ActivePageTracker`, and `UrlSessionStore`. React gets a derived snapshot and calls application commands.
5. `SidePanelView`: controlled view. It renders Current Page Card and Draft Prompt from the snapshot and sends user intent back to the controller.

This follows `docs/ARCHITECTURE.md`: UI renders derived state, the extension owns URL Session state and active-page selection, and bridge/provider lifecycle stays behind the bridge boundary.

```ts
describe("resolvePageIdentity", () => {
  it("prefers_canonical_url_when_present", () => {
    expect(resolvePageIdentity({
      url: "https://example.com/current?x=1",
      canonicalUrl: "https://example.com/canonical?utm_source=newsletter#section"
    })).toMatchObject({ status: "ready", pageKey: "https://example.com/canonical" });
  });

  it("falls_back_to_current_url_without_hash", () => {
    expect(resolvePageIdentity({ url: "https://example.com/article#comments" })).toMatchObject({
      status: "ready",
      pageKey: "https://example.com/article"
    });
  });

  it("strips_common_tracking_parameters_case_insensitively", () => {
    expect(resolvePageIdentity({
      url: "https://example.com/article?UTM_Source=x&fbclid=y&GCLID=z&id=42"
    })).toMatchObject({ pageKey: "https://example.com/article?id=42" });
  });

  it("preserves_non_tracking_query_parameters_and_stabilizes_order", () => {
    const first = resolvePageIdentity({ url: "https://example.com/article?b=2&a=1" });
    const second = resolvePageIdentity({ url: "https://example.com/article?a=1&b=2" });
    expect(first).toMatchObject({ status: "ready" });
    expect(second).toMatchObject({ status: "ready" });
    if (first.status !== "ready" || second.status !== "ready") throw new Error("expected ready pages");
    expect(first.pageKey).toBe(second.pageKey);
  });

  it("falls_back_to_current_url_when_canonical_url_is_invalid", () => {
    expect(resolvePageIdentity({
      url: "https://example.com/current",
      canonicalUrl: "not a url"
    })).toMatchObject({ pageKey: "https://example.com/current" });
  });

  it("keeps_distinct_page_keys_for_distinct_non_tracking_queries", () => {
    const first = resolvePageIdentity({ url: "https://example.com/article?id=1" });
    const second = resolvePageIdentity({ url: "https://example.com/article?id=2" });
    expect(first).toMatchObject({ status: "ready" });
    expect(second).toMatchObject({ status: "ready" });
    if (first.status !== "ready" || second.status !== "ready") throw new Error("expected ready pages");
    expect(first.pageKey).not.toBe(second.pageKey);
  });
});
```

Assertions:

- Canonical URL wins over current URL when canonical is a valid `http:` or `https:` URL.
- Fragments are removed from canonical and current URL inputs.
- Tracking parameter names are matched case-insensitively.
- Strip any parameter beginning with `utm_`.
- Strip exact names: `fbclid`, `gclid`, `dclid`, `gbraid`, `wbraid`, `msclkid`, `yclid`, `igshid`, `mc_cid`, `mc_eid`, `_hsenc`, `_hsmi`.
- Preserve non-tracking parameters and make equivalent query ordering produce the same Page Key.
- Preserve distinct non-tracking query values as distinct Page Keys.

## Performance Considerations

- URL Sessions are in memory only and expected to be small for V1.
- Active tab tracking should debounce nothing initially unless tests expose event storms; stale-read protection is required because async tab reads can complete out of order.
- Do not eagerly start provider sessions on tab switch. Start remains lazy on first send through the session coordinator.
- Do not create URL Sessions for unsupported or missing active page states.
- Do not passively inspect DOM or inject scripts to find canonical URL.

## Migration Notes

- No persisted data migration is needed.
- Existing single-session tests must be updated to select an active page in their harness.
- Existing bridge protocol and bridge session manager behavior should remain unchanged.
- Existing `New Chat` UI wiring remains, but it becomes active URL Session scoped through `UrlSessionStore`.

## Risks And Review Targets

- Duplicate ownership of Transcript between `UrlSessionStore` and `BridgeSessionCoordinator` would violate the architecture. The store should hold coordinators and derive transcript from them, not copy transcript state.
- React local Draft Prompt state would break URL Session draft restoration. The view must be controlled.
- Reading canonical URL by injecting a script during tab switch would violate explicit capture rules. Canonical preference must be supported by the normalizer and future capture path, not passive DOM reads.
- Reusing one `BridgeSessionCoordinator` across all Page Keys would leak provider sessions. There must be one coordinator per URL Session.
- Calling `session.start` again every time a user returns to a page would replace the bridge provider session. Returning to a Page Key must reuse the existing Client Session ID and coordinator state.
- Marking only the active session disconnected on bridge disconnect leaves inactive sessions with stale provider state. Disconnect handling must cover all started/starting sessions.
- Unsupported active pages must not keep showing the previous page's transcript.
