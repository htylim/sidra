## Desired End State

The composer options button opens a compact popover anchored above the composer. The popover contains one checkbox/toggle labeled `Send Full DOM`. The popover-open state is local view state. The actual capture mode is session state exposed by `SidePanelSnapshot.activeSession.captureMode` and changed through a controller command, because capture mode affects application behavior. Define the shared capture-mode type in a neutral extension domain file such as `apps/extension/src/capture-mode.ts`, not in React and not in `CaptureService`.

New URL sessions default to readable capture mode. When `Send Full DOM` is off, `Capture + Send` keeps the current readable behavior. When it is on, `Capture + Send` sends a `PageContext` with `kind: "full_dom"`, metadata, full document HTML, and `htmlLength`. It does not send readable text or `extractionMethod` in the same context.

Settings add a separate `domContentLimitCharacters` value. This setting loads from the same `SettingsStore`, validates independently, applies live to later capture attempts, and does not appear in the prompt popover. Recommended V1 constants:

- `DEFAULT_DOM_CONTENT_LIMIT_CHARACTERS = 300_000`
- `MIN_DOM_CONTENT_LIMIT_CHARACTERS = 1_000`
- `MAX_DOM_CONTENT_LIMIT_CHARACTERS = 750_000`

If full DOM HTML exceeds the DOM limit, Sidra sends metadata only with a DOM-specific reason. It does not send partial HTML. The page card shows `Full DOM skipped: too large`. The transcript marker says `Full DOM skipped; content too large`. Raw DOM never appears in the transcript.

Capture mode is resolved for the URL session identified by the captured page identity, not for a stale active-page tracker snapshot. `Capture + Send` must first read the active tab at click time and resolve the captured/canonical page identity, select or create that URL session, then build the page context using that captured session's capture mode. If canonical URL resolution creates a new URL session for the same click, carry over the pre-capture active session's capture mode for that first send and persist it on the new canonical session. This preserves URL-session scoping when the active-page tracker is stale, while still honoring the user's just-selected `Send Full DOM` toggle when canonical normalization moves the send into a new session.

The bridge prompt wrapper continues to treat the full DOM payload as untrusted reference material. The bridge hard payload ceiling remains a separate safety guard. If a full DOM context is under the DOM character limit but still exceeds the serialized Native Messaging payload byte limit, the extension/bridge payload-too-large behavior remains separate from DOM skipped-context handling.

### Key Discoveries

- Full DOM capture does not require a new content-script read path because `CapturedTabDocument.html` already captures `document.documentElement.outerHTML`: `apps/extension/src/capture-script.ts:33`. It may still be worth splitting document snapshot reading from page-context construction so the controller can select the captured URL session before capture mode is applied.
- Full DOM does need a new protocol variant. Using `metadata_only` for successful full DOM sends would erase the fact that DOM content was included, and overloading `readable.text` would violate mutual exclusion.
- Capture mode should not live only in React. The view can own whether the popover is open, but `UrlSessionStore` should own `captureMode` because URL sessions already own draft/context behavior and issue 004-bis requires send defaults to be URL-session scoped. The type itself should live in a small neutral domain file so `UrlSessionStore`, `SidePanelController`, `CaptureService`, and `SidePanelView` can share it without importing through the capture-service implementation boundary.
- The popover must not become a settings UI. The PRD says DOM size limit is controlled by extension settings, not the prompt popover: `docs/prd-v1.md:166`.

## What We're NOT Doing

- No passive DOM capture, background capture, recurring capture, or pre-send DOM preview.
- No partial DOM truncation.
- No sending readable text and full DOM in the same `PageContext`.
- No settings/options page UI.
- No quick action changes.
- No provider or Codex SDK integration changes.
- No token estimation or provider-specific context budgeting.
- No prompt or captured-content logging.
- No bridge protocol state in React.
- No change to issue 004-bis plain-send default behavior beyond making capture mode available to `Capture + Send`.

## Implementation Approach

Use vertical TDD at the owning boundaries:

1. Grow protocol, transcript markers, context state, and bridge prompt tests so full DOM can be represented safely before capture uses it.
2. Add `domContentLimitCharacters` to the settings boundary with independent validation and live updates.
3. Add capture-mode-aware context construction to `CaptureService`.
4. Add session-scoped capture mode to `UrlSessionStore` and `SidePanelController`, then route `Capture + Send` through that mode.
5. Build the compact popover in `SidePanelView` as a view over controller snapshot state.
6. Run focused package tests, type checks, and a manual side-panel smoke pass.

## Testing Strategy

This section complements phase-local verification. It does not replace it.

### Unit Tests

- Protocol parser accepts and rejects full DOM context shapes.
- Settings parsing validates DOM limit independently from readable limit.
- Capture service builds full DOM or skipped metadata-only context from mode and settings.
- URL sessions retain capture mode per session and reset on `New Chat`.
- Transcript marker selection never includes raw HTML.

### Integration Tests

- Controller routes `Capture + Send` through the active session capture mode.
- Controller resolves the captured/canonical URL session before choosing capture mode.
- Controller uses live settings for later full-DOM captures.
- View popover controls callback wiring without sending prompts.

### Manual Testing Steps

1. Read `docs/MANUAL-E2E-RUNBOOK.md`.
2. Load the extension with the local bridge.
3. Open a text-heavy page and send with default settings. Confirm readable context marker remains unchanged.
4. Open prompt options, enable `Send Full DOM`, send again, and confirm full DOM marker.
5. Lower DOM limit through local extension storage or the runbook fixture, send with `Send Full DOM`, and confirm skipped marker with no raw DOM in transcript.
6. Switch tabs/URLs and confirm each URL session keeps its own capture mode.

## Performance Considerations

Full DOM capture can create large strings. Do not clone or parse DOM in full-DOM mode after the content script returns `document.documentElement.outerHTML`. Check `html.length` before constructing `session.send` payloads that include HTML. Keep the existing serialized payload preflight as the last extension-side guard before Native Messaging.

## Migration Notes

Existing stored settings may only contain `readableContentLimitCharacters`. Treat missing `domContentLimitCharacters` as default. Parse settings independently so old settings remain valid.
