Goal

Add two subtle action buttons to transcript message bubbles. The top button reads that bubble aloud with OpenAI Speech. The lower button copies that bubble text to the clipboard.

The speech button starts as Play. While speech is loading or playing, it uses a Pause icon and a darker active state. The user can pause active audio, resume paused audio, or start another bubble, which stops the current bubble first.

The settings page must expose every user-facing control for this feature, including OpenAI API key management. The API key must be stored by the local bridge in an OS secret store, not in Chrome extension settings.

Non-goals

  • No chrome.tts. The quality is not acceptable.
  • No Realtime API in V1. This is one-way click-to-read playback, not a live voice conversation.
  • No API key persistence in chrome.storage, local storage, logs, test snapshots, or plain config files.
  • No automatic read-aloud, prefetching, or background synthesis.
  • No custom voices or voice cloning.
  • No persistent audio cache.
  • No controls on status chips, permission cards, setup panels, activity disclosures without a response body, or page cards.

Research Findings

OpenAI API access and billing

  • OpenAI says ChatGPT subscriptions and the API platform use separate billing systems. API use is billed separately from ChatGPT Pro, Plus, Business, Enterprise, and Edu. Source: OpenAI Help billing article and OpenAI API pricing.
  • Sidra settings should let the user save, test, replace, and remove the OpenAI API key. The extension sends the key to the local bridge only during an explicit save or test action. The bridge stores the key in the OS secret store and never returns the raw key to the extension.
  • The bridge may also read SIDRA_OPENAI_API_KEY and OPENAI_API_KEY as fallbacks. A settings-managed key takes precedence over environment keys.
  • A missing speech API key must not block normal chat bridge readiness. Chat can still work. A speech click returns a scoped speech.error with openai_api_key_missing.

Speech API choice

  • The OpenAI Speech guide documents gpt-4o-mini-tts for text-to-speech and recommends marin or cedar for best quality. Source: OpenAI text-to-speech guide.
  • The Speech API reference limits a single input to 4096 characters. Longer bubbles must be split into ordered speech segments. Source: Create speech API reference.
  • The Speech API supports streamed audio. Use streaming for faster first audio than waiting for a full article audio file.
  • The Realtime API can be faster for live conversational audio, but it adds session state, microphone-oriented semantics, and higher complexity. It is not the right first implementation for static bubble readback.

UI and accessibility research

  • Use real buttons. WAI-ARIA says buttons need accessible names, Space and Enter activation, and disabled state when unavailable. Source: WAI-ARIA Authoring Practices button pattern.
  • If the play/pause label changes by state, do not also model it as a fixed-label toggle button. The accessible label should describe the next action: Read, Pause, or Resume.
  • Use a target size of at least 24 by 24 CSS pixels. Use 28 by 28 CSS pixels here so two stacked buttons remain compact but not fragile. Source: WCAG 2.2 target size minimum.
  • Icon-only buttons need an accessible name and visible affordance. Sidra already records this in docs/UI-GUIDELINES.md.
  • Tooltip behavior must not be required to operate the feature. Use aria-label and title for V1 rather than a custom tooltip. Source: WAI-ARIA tooltip pattern.
  • Clipboard writes are async, require permission from the user agent, and can fail. The copy button must show a transient success or failure state. Source: MDN Clipboard.writeText.
UI decision: show the action rail only on message hover, message focus-within, or active audio state. Keep the rail in the DOM so keyboard users can tab to it. Use opacity and color changes only; do not move the bubble.

Current State

  • apps/extension/src/transcript-view.tsx renders transcript entries and currently renders user and assistant message bubbles without per-bubble action controls.
  • apps/extension/src/transcript.ts stores plain user text and assistant text alongside assistant markdown. Assistant turns do not currently get stable ids when they are created from stream deltas.
  • apps/extension/src/settings-store.ts owns persisted Sidra settings. It already has grouped save methods for quick actions and transcript font sizes.
  • apps/extension/src/options-page-view.tsx renders the settings page and has validation patterns for number inputs, textareas, dirty section flags, save errors, and disabled save state.
  • apps/extension/src/bridge/connection.ts owns Native Messaging connection state and validates all bridge messages through parseBridgeToExtension.
  • apps/extension/src/side-panel-controller.ts composes settings, page state, bridge connection, and URL-scoped sessions into the side-panel snapshot.
  • apps/bridge/src/index.ts validates incoming Native Messaging messages and dispatches them to bridge managers.
  • apps/bridge/src/session-manager.ts owns Codex provider sessions. Do not add OpenAI Speech behavior there. Add a separate speech manager.
  • packages/protocol/src/index.ts owns protocol types and runtime validation. It must grow before extension or bridge code depends on new speech messages.

Architecture

Ownership decision

Concern Owner Decision
OpenAI API key and HTTP call apps/bridge The extension may hold the key only as transient form input during Save or Test. The bridge owns persistence and all OpenAI HTTP calls.
Speech secret storage apps/bridge/src/speech-credential-store.ts Store settings-managed keys in the OS secret store. On macOS use Keychain service com.sidra.openai, account default.
Speech message contract packages/protocol Add protocol v3 speech commands and bridge events with runtime validation.
Speech request lifecycle and cancellation apps/bridge/src/speech-synthesis-manager.ts Separate from BridgeSessionManager. It is not a Codex provider session.
Audio playback state apps/extension/src/transcript-speech-controller.ts Extension-side application boundary. React renders a snapshot and sends intents.
Bubble action rail apps/extension/src/transcript-view.tsx View-only controls. It receives state and callbacks; it does not own bridge protocol details.
Copy to clipboard apps/extension Use a small clipboard gateway or hook so tests do not depend on global browser state.

Runtime flow

  1. User hovers or tabs into a transcript message row.
  2. The action rail becomes visible. Buttons remain in the DOM before hover.
  3. User clicks Play.
  4. TranscriptSpeechController normalizes the bubble text, validates settings, stops any active bubble, and posts speech.synthesize through BridgeConnection.
  5. createBridge validates the message through parseExtensionToBridge and dispatches to SpeechSynthesisManager.
  6. SpeechSynthesisManager asks SpeechCredentialStore for the API key. Keychain wins over environment fallback.
  7. SpeechSynthesisManager splits text into 4096-character-or-smaller segments and streams each segment through the OpenAI Speech API.
  8. The bridge emits speech.started, ordered speech.chunk messages, and speech.done. Chunk messages must be small enough to stay far below the native messaging hard payload limit.
  9. The extension validates bridge events through parseBridgeToExtension, appends audio chunks to the playback gateway, and starts playback after the first playable chunk.
  10. Pause pauses local audio. Resume continues buffered audio. Cancel during loading posts speech.cancel.

Text normalization

  • Add apps/extension/src/transcript-message-text.ts.
  • It returns normalized plain text for speech and copy.
  • User messages return entry.text.trim().
  • Quick-action user messages are excluded from V1 action rails even though they have stored prompt text.
  • Assistant messages strip common Markdown syntax, preserve paragraph and list breaks, preserve code block text, remove image syntax, and turn Markdown links into visible link text.
  • Do not use rendered HTML as the source of truth. Use a pure helper with tests.

Stable message ids

Add stable ids to assistant turns. The action rail and speech state must not depend on array indexes. BridgeSessionCoordinator already creates ids for user prompts. It must also allocate one assistant turn id for each sent turn before the first assistant delta can create the assistant entry.

Validation rules

  • Reject unknown fields on all new message variants.
  • Require non-empty clientSessionId, requestId, and entryId where applicable.
  • Require non-empty apiKey for speech.credentials.save.
  • Never include raw apiKey in any bridge-to-extension message.
  • Require text.trim().length > 0.
  • Reject text over 50000 characters at protocol parse time.
  • Allow only model gpt-4o-mini-tts in V1.
  • Allow only response format mp3 in V1.
  • Validate voice against the built-in allowlist.
  • Validate speed from 0.25 to 4.0.
  • Validate instructions length at 4096 characters in protocol, even though the settings UI caps at 600.
  • Validate speech.chunk.base64Audio as non-empty base64 text and keep each chunk message below 128 KiB serialized.

UI and Accessibility

Action rail layout

  • Add a TranscriptMessageFrame wrapper around actionable message bubbles.
  • The wrapper reserves a 32px right-side action rail so the bubble never shifts when controls appear.
  • The rail is top-aligned with the message bubble and stacks two 28px square icon buttons with a 6px gap.
  • The rail sits to the right of the bubble, matching the screenshot direction.
  • Reduce the effective message max width so the rail never overlaps the scrollbar or composer.
  • On very narrow widths, keep the rail inside the row but preserve at least 24px target size and 4px spacing.

Visual states

  • Default rail opacity: 0.
  • Row hover, row focus-within, copied state, failed state, loading state, playing state, and paused state opacity: 1.
  • The action rail sets aria-live="off". Speech and copy status announcements use a separate small live region so the chat transcript live region does not announce every icon state change.
  • Idle button color should match the muted status-card color family: border #d8e1dd, text #62716d, background #ffffff with low opacity.
  • Hover and focus-visible darken to #52615d text and #eef3f1 background.
  • Active speech uses darker teal-neutral styling: text #075e56, border #83b8ae, background #eef8f5.
  • Failure uses the existing warm error family from setup/error states.
  • No layout property changes on hover, active, copied, failed, loading, playing, or paused.
