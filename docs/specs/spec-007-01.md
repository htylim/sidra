# Empty-State Quick Actions Implementation Plan

## Status

Complete

## Overview

Implement issue 007: empty URL sessions show configurable quick actions, and the V1 default `Summarize this page` quick action sends its configured prompt through the existing `Capture + Send` path.

Quick actions are extension-owned application behavior. Settings owns configuration. `SidePanelController` owns the view snapshot and command routing. React only renders configured actions and forwards user intent.

## Current State Analysis

Issue 007 requires a centered empty state, one default `Summarize this page` quick action with the PRD prompt, settings-backed enable/disable and add/remove behavior, quick-action click behavior using `Capture + Send`, and tests for defaults, disabled state, custom actions, empty-session-only visibility, and capture mode: `docs/issues/issue-007.md:13`.

The current code has a hard-coded empty-state action, but it does not use quick-action settings and it does not send:

- `SidePanelView` renders the empty state when the active session transcript is empty: `apps/extension/src/side-panel-view.tsx:65`.
- The empty-state helper text says `Use the action below or ask your own question.`, while the PRD requires `Use the actions below or ask your own question.`: `apps/extension/src/side-panel-view.tsx:69`, `docs/prd-v1.md:368`.
- The hard-coded `Summarize this page` button only calls `onDraftPromptChange("Summarize this page")`: `apps/extension/src/side-panel-view.tsx:70`.
- `SidePanelView` already sends normal composer submits through `onCaptureAndSend`: `apps/extension/src/side-panel-view.tsx:25`, `apps/extension/src/side-panel-view.tsx:31`, `apps/extension/src/side-panel-view.tsx:124`.
- `SidePanelController.captureAndSend()` trims prompts, blocks unusable chat, captures the active tab at click time, resolves the captured URL session, uses that session's `captureMode`, builds page context, and sends through `sendPromptWithContext`: `apps/extension/src/side-panel-controller.ts:173`.
- `UrlSessionStore` owns URL-session draft, context state, transcript, and capture mode: `apps/extension/src/url-session-store.ts:31`.
- New URL sessions default to readable capture mode: `apps/extension/src/url-session-store.ts:205`.
- `SettingsStore` already owns storage loading, validation, defaults, local-storage live changes, and snapshot cloning for readable and DOM limits: `apps/extension/src/settings-store.ts:1`.
- Production Chrome settings access is intentionally isolated to `settings-store.ts`: `apps/extension/src/settings-store.ts:114`.

The PRD says quick actions appear only in empty sessions, use `Capture + Send` by default, are configurable in settings, and ship with one default action: `docs/prd-v1.md:351`. The required default label and prompt are defined in `docs/prd-v1.md:376`.

## Desired End State

`SettingsStore` exposes a validated quick-action configuration:

```ts
export type QuickAction = {
  id: string;
  label: string;
  prompt: string;
};

export type QuickActionsSettings = {
  enabled: boolean;
  actions: QuickAction[];
};
```

The default settings snapshot includes:

- `quickActions.enabled: true`
- one action with label `Summarize this page`
- the exact PRD default summary prompt

Stored settings may enable or disable quick actions and replace the action list. Each stored action must have a non-empty `label` and non-empty `prompt` after trimming. Invalid action entries are ignored. Action IDs are internal identifiers. Invalid or duplicate stored IDs are replaced with deterministic unique IDs during parsing. An explicit empty action list is valid and renders no quick-action buttons.

`SidePanelSnapshot` exposes derived visible quick actions for the active session:

```ts
export type VisibleQuickAction = {
  id: string;
  label: string;
};
```

The snapshot must not expose quick-action prompts. React only needs IDs and labels. `SidePanelController.sendQuickAction(actionId)` resolves the current configured prompt internally and delegates to `captureAndSend(prompt)`.

When settings have finished their initial load, the bridge is ready, the active page is capturable, quick actions are enabled, and the active URL session transcript is empty, the view renders:

- centered icon
- heading `Ask anything about this page`
- helper text `Use the actions below or ask your own question.`
- quick-action grid

Clicking a quick action sends that action's configured prompt through `SidePanelController.captureAndSend()`. It does not first write the prompt into the draft. The existing capture path decides readable versus full-DOM based on the active URL session's `captureMode`, so quick actions honor the user's current capture mode.

Quick actions are hidden when:

- bridge setup blocks chat
- the active page is unsupported
- initial settings have not loaded yet
- quick actions are disabled
- the active URL session already has transcript entries

Issue 007 also needs an actual settings surface for enable/disable and add/remove. Add a minimal extension options page registered through `manifest.options_ui`. The side-panel header gear opens that options page through the controller boundary. The options page is the only UI that writes quick-action settings in this issue.

## What We're NOT Doing

- No quick-action edit UI inside the side panel. Configuration lives in the extension options page.
- No new bridge protocol.
- No plain `Send` mode changes from issue 004-bis.
- No passive capture.
- No quick-action icons beyond the simple V1 button icon/label styling.
- No persistence of URL-session transcripts or drafts beyond the current in-memory behavior.
- No provider selection UI, even though the PRD says provider settings exist later.

## Implementation Approach

Add quick-action settings at the existing settings boundary first. Then subscribe the controller to settings changes and derive view-ready quick actions in `SidePanelSnapshot` after initial settings load. Replace the hard-coded empty-state button with a grid that calls a new quick-action intent. Add the minimal options page and header settings command last, after the settings contract is stable.

Keep the prompt string as a named exported constant in the settings/domain boundary so future settings UI can use the same source of truth. Tests must include one independent literal assertion for the default prompt so the test cannot pass by importing the same broken constant it is meant to verify.

Integration-style tests:

2. Confirm the empty state shows `Ask anything about this page`, `Use the actions below or ask your own question.`, and `Summarize this page`.

4. Confirm a context marker appears and the user prompt is the full PRD prompt, not only the button label.

7. Disable quick actions in local storage.

## Performance Considerations

Quick-action settings are small. Snapshot cloning should clone the quick-action array but must not deep-clone unrelated session transcript data. `sendQuickAction()` should only call `captureAndSend()` after the user clicks; no page content is captured during rendering or settings updates.
