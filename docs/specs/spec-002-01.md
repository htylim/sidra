# Issue 002 Bridge Availability And Retry Implementation Plan

## Overview

Implement bridge availability as a first-class side-panel state. The side panel must show the chat experience only after the Native Messaging bridge is usable, block prompting when the bridge is unavailable or reports an error, and reconnect through Retry without reloading the side panel.

This plan keeps bridge readiness and Native Messaging behavior inside the extension bridge boundary. React renders a derived snapshot and calls application commands only.

## Current State Analysis

Issue #002 is the foundation for the later URL session, capture, streaming, cancel, reset, permission, heartbeat, and Codex-provider issues. The issue asks for connected, unavailable, invalid-message, and retry behavior.

Current implementation already has useful boundaries:

- `apps/extension/src/bridge/connection.ts:35` owns the extension Native Messaging port lifecycle and invokes the injected Native Messaging connector. The Chrome-specific connector is currently supplied by the controller composition factory.
- `apps/extension/src/bridge/connection.ts:109` validates inbound bridge messages with `parseBridgeToExtension`.
- `apps/extension/src/bridge/session-coordinator.ts:40` owns session-start ordering and queues prompts until `session.started`.
- `apps/extension/src/side-panel-controller.ts:41` composes `BridgeConnection` and `BridgeSessionCoordinator` into the view-facing API.
- `apps/bridge/src/native-messaging.ts:19` emits `bridge.ready` when the native bridge starts.
- `apps/bridge/src/native-messaging.ts:26` and `apps/bridge/src/index.ts:18` emit parser-backed `bridge.error` messages for invalid JSON or invalid extension protocol messages.
- `packages/protocol/src/index.ts:62` and `packages/protocol/src/index.ts:90` define runtime validation for both directions of the Native Messaging boundary.

Important gaps:

- `BridgeConnection` connects lazily from `post()` at `apps/extension/src/bridge/connection.ts:64`. The side panel does not proactively detect bridge availability when it opens.
- `BridgeConnection.retry()` at `apps/extension/src/bridge/connection.ts:79` calls `ensurePort()`, but `ensurePort()` returns the existing port when one exists. Retry does not reliably replace a stale or erroring port.
- `SidePanelView` renders empty chat and enabled composer controls regardless of bridge state at `apps/extension/src/side-panel-view.tsx:40` and `apps/extension/src/side-panel-view.tsx:59`.
- `SidePanelController.sendPrompt()` at `apps/extension/src/side-panel-controller.ts:101` always delegates to the coordinator. It does not gate sends on bridge availability.
- Invalid inbound bridge messages currently set `setupError` in `BridgeConnection` at `apps/extension/src/bridge/connection.ts:109`, but the view does not turn that into a blocking setup state.

Product and architecture constraints:

- `docs/ARCHITECTURE.md:13` assigns bridge readiness, reconnect, disconnect behavior, and session-start coordination to `apps/extension/src/bridge`.
- `docs/ARCHITECTURE.md:40` says UI code should call application-level commands such as `sendPrompt` and `retryBridge`.
- `docs/prd-v1.md:509` requires a blocking setup/error panel, disabled prompt input, and Retry when the bridge is unavailable.
- This codebase is learning-oriented. New state should use named TypeScript variants instead of loose booleans.

## Desired End State

The side panel opens by attempting to connect to the Native Messaging bridge. Until the bridge is usable, the chat area shows a setup/checking/error panel and the prompt composer cannot send. When the current port emits `bridge.ready`, `BridgeConnection` marks bridge availability as ready and the connected chat experience becomes visible. If the bridge is unavailable, emits `bridge.error`, sends an invalid protocol message, or disconnects at any time, the side panel blocks prompting and exposes Retry. Retry opens a fresh Native Messaging port and can recover without reloading the side panel.

The bridge and extension continue to reject unknown or invalid protocol messages through `packages/protocol` parsers. Bridge-side invalid protocol messages surface as `bridge.error` with `code: "invalid_message"`. Extension-side invalid inbound messages do not fabricate a protocol message; they set bridge availability to an error state with `code: "invalid_message"` and do not notify protocol subscribers.

### Key Discoveries

- The right extension owner already exists: `BridgeConnection` owns port lifecycle, readiness, retry, disconnect, raw protocol validation, and transport state.
- The right application owner already exists: `SidePanelController` owns the view-facing snapshot and should expose product terms such as `availability` and `canUseChat`.
- The right bridge owner already exists: `createBridge()` parses extension messages before dispatching them to `BridgeSessionManager`.
- The UI must stay dumb. It should render snapshot states and call `onRetryBridge`, not inspect Chrome port behavior or protocol sequencing.

## What We're NOT Doing

- Not installing Native Messaging host manifests.
- Not adding URL-scoped sessions.
- Not adding capture, context markers, settings, or size limits.
- Not adding Codex provider integration.
- Not adding heartbeat cleanup.
- Not adding persistent chat history.
- Not changing provider session lifecycle. `session.started` remains only the bridge response to `session.start` and stays owned by `BridgeSessionCoordinator`.
- Not moving Native Messaging or protocol state machines into React components.

## Implementation Approach

Model bridge availability explicitly at the extension bridge boundary, derive UI-ready state in the side-panel controller, and then render the blocking panel from the derived snapshot.

Use a small discriminated union for bridge availability. Keep the current `connected` and `ready` booleans available if needed for compatibility, but do not make new behavior depend on interpreting several independent booleans.

Suggested state shape:

```ts
export type BridgeAvailability =
  | { status: "checking"; message: string }
  | { status: "ready" }
  | { status: "unavailable"; message: string }
  | { status: "error"; message: string; code?: string };
```

The controller can derive:

```ts
type SidePanelBridgeSnapshot = {
  availability: BridgeAvailability;
  // Compatibility fields may remain during migration, but new gating logic must
  // use availability and canUseChat instead of recomputing from these booleans.
  connected: boolean;
  ready: boolean;
  setupError?: string;
  canUseChat: boolean;
};
```

`canUseChat` is true only when `availability.status === "ready"`. `BridgeConnection` enters `ready` only from `bridge.ready`. Blocking states such as `unavailable`, `error`, and disconnect always clear readiness and override any previous ready state.

## Behavior Matrix

Every row below must be covered by at least one named test in the phases.

| State or event | Expected side-panel behavior | Covered by |
| --- | --- | --- |
| Side panel opens and a port is created but no `bridge.ready` has arrived | Show checking/setup panel, disable composer | `connects explicitly and enters checking before bridge.ready`, `renders checking setup panel and disables prompt controls` |
| `bridge.ready` arrives | Show connected chat experience, enable composer | `marks bridge.ready as usable chat availability`, `renders empty chat only when chat is usable` |
| `connectNative` throws | Show retryable unavailable panel, disable composer | `records unavailable state when connectNative throws`, `does not send prompts while bridge is unavailable` |
| Native port disconnects before readiness | Show retryable unavailable panel | `records unavailable state when the startup port disconnects before readiness` |
| Native port disconnects after `bridge.ready` but before any session starts | Show retryable unavailable panel and keep composer blocked | `records unavailable state when a ready idle port disconnects` |
| Native port disconnects after session activity | Existing transcript is preserved with bridge disconnected status, composer blocks new sends until retry | `blocks new prompts after bridge disconnect until retry succeeds` |
| Bridge emits `bridge.error` | Show retryable error panel, disable composer | `surfaces bridge.error as blocking availability state`, `renders bridge error panel with retry` |
| Bridge sends an invalid inbound message | Parser rejects it, protocol subscribers are not notified, UI blocks with `availability.status === "error"` and `code: "invalid_message"` | `blocks on invalid inbound bridge messages before notifying subscribers` |
| User clicks Retry after failure | Fresh port is opened without side-panel reload | `retry opens a fresh native port after unavailable state`, `retryBridge reconnects without recreating the controller` |
| Retry succeeds with `bridge.ready` | Connected chat returns and composer enables | `retryBridge returns to usable chat after bridge.ready` |
| Unknown or invalid extension-to-bridge message reaches bridge | Bridge emits `bridge.error` with `code: "invalid_message"` | `emits invalid_message bridge.error for unknown extension commands` |

| --- | --- |
| Side panel shows connected chat experience only after `bridge.ready` | Phase 1 and Phase 2 availability state, Phase 3 rendering tests |
| Bridge unavailable or `bridge.error` shows blocking error panel and disables prompt input | Phase 2 prompt gating, Phase 3 blocking UI |
| Retry attempts reconnect without side-panel reload | Phase 1 retry semantics, Phase 2 controller retry tests, Phase 3 UI Retry |
| Bridge and extension reject unknown or invalid protocol messages with parser-backed errors | Phase 1 invalid inbound extension handling, Phase 4 bridge/protocol tests |
| Tests cover connected, unavailable, invalid-message, and retry states | Behavior matrix and all four phases |

## Risks And Mitigations

- Risk: Proactive connection makes the existing e2e smoke test fail on machines without a Native Messaging host. Mitigation: update the smoke test to expect the setup panel when the host is missing.
- Risk: Retry reuses a stale connected port and cannot recover from setup errors. Mitigation: Phase 1 requires fresh-port retry tests and stale-port guards.
- Risk: View code starts interpreting low-level bridge details. Mitigation: Phase 2 exposes `canUseChat` and `availability` in product terms; Phase 2 boundary test protects the React entry point.
- Risk: Existing tests assume prompt send is allowed before readiness. Mitigation: update controller tests to reflect issue #002 behavior and keep session-coordinator tests for lower-level session ordering.
- Risk: Issue #002 mentions successful session start as an alternate signal, but the current bridge emits `bridge.ready` before any session start can be posted. Mitigation: keep `session.started` as session lifecycle behavior owned by `BridgeSessionCoordinator`, and do not add an unreachable bridge-readiness fallback.
