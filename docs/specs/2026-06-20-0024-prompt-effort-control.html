Implementation Plan - Spec 0024 - 2026-06-20

Prompt Effort Control

Add a Sidra control that lets the user choose the Codex reasoning effort used for prompts. The selected effort must travel as structured data from the extension to the bridge and into Codex App Server turn/start.

Overview Current State Desired State Decisions Flow Phase 1 Phase 2 Phase 3 Phase 4 Testing References
Status

Draft ready for implementation.

Primary Owners

packages/protocol, apps/extension, apps/bridge.

Effort Values

low, medium, high, xhigh. Default medium.

Out Of Scope

Model picker, dynamic model catalog UI, prompt-string hacks, provider-independent effort translation.

Overview

Sidra currently sends only the prompt text and optional page context across the Native Messaging boundary. Codex App Server supports a structured effort parameter on turn/start. That parameter applies to the current turn and then becomes sticky for subsequent turns in the thread, so Sidra must send the selected effort on every production turn. This plan adds a compact side-panel prompt effort control, persists the chosen value as a Sidra setting, and samples that value into each prompt submission.

The work is split so protocol validation comes first, extension UI/settings come second, bridge/provider propagation comes third, and durable docs/manual validation come last.

Current State Analysis

Key Discoveries

  • packages/protocol owns Native Messaging message types and runtime validation; new message shape must start there before UI or bridge code depends on it. See docs/ARCHITECTURE.md:23-25 and docs/ARCHITECTURE.md:107.
  • session.send currently allows only type, version, clientSessionId, prompt, and optional pageContext. Extra fields are rejected. See packages/protocol/src/index.ts:87-100 and packages/protocol/src/index.ts:263-265.
  • BridgeSessionCoordinator queues prompts before session.started, then builds session.send in one place. See apps/extension/src/bridge/session-coordinator.ts:62-81, apps/extension/src/bridge/session-coordinator.ts:457-497.
  • SettingsStore owns persisted settings, live storage updates, and cloning/normalization. See apps/extension/src/settings-store.ts:3-11, apps/extension/src/settings-store.ts:90, and apps/extension/src/settings-store.ts:202-235.
  • The options page already uses draft state, dirty refs, validation, and one save flow. See apps/extension/src/options-page-view.tsx:78-155 and apps/extension/src/options-page-view.tsx:332-376.
  • The side-panel controller already reads settings live and samples them during send/capture actions. See apps/extension/src/side-panel-controller.ts:216-229 and apps/extension/src/side-panel-controller.ts:235-305.
  • captureAndSendCommand awaits page capture before posting the prompt, so effort must be snapshotted before the first capture await. See apps/extension/src/side-panel-controller.ts:235-261.
  • UrlSessionStore sits between the side-panel controller and BridgeSessionCoordinator for both plain and capture sends. It currently accepts only prompt text or prompt plus page context, so prompt effort must be added there too. See apps/extension/src/url-session-store.ts:61-65, apps/extension/src/url-session-store.ts:154-177.
  • BridgeSessionManager converts validated bridge input to provider-neutral AgentSendInput. It currently carries only formatted prompt text and display-title source. See apps/bridge/src/session-manager.ts:22-25 and apps/bridge/src/session-manager.ts:147-154.
  • The Codex provider currently omits effort from turn/start. See apps/bridge/src/codex-app-server-provider.ts:147-154.
  • The existing Codex App Server investigation already models turn/start with model and effort. See docs/codex-sdk-investigation.md:422-434.
  • Local Codex CLI 0.141.0 generated TypeScript bindings show TurnStartParams.effort?: ReasoningEffort | null and model metadata with supportedReasoningEfforts. Research command: codex app-server generate-ts --out /private/tmp/codex-app-server-ts --experimental.
Raw local Codex protocol findings

Generated file /private/tmp/codex-app-server-ts/v2/TurnStartParams.ts declares effort?: ReasoningEffort | null with comment "Override the reasoning effort for this turn and subsequent turns." That stickiness means Sidra cannot rely on sending effort only when it changes. Every accepted production prompt must carry the selected effort, including medium. Generated file /private/tmp/codex-app-server-ts/v2/Model.ts declares supportedReasoningEfforts and defaultReasoningEffort. Generated ReasoningEffort is typed as string, so Sidra should constrain its own public protocol to the currently exposed Codex values rather than accepting arbitrary strings from the extension.

Desired End State

  • The side-panel composer exposes a compact Prompt effort select with Low, Medium, High, and Extra high.
  • The selected effort is stored in SettingsStore, defaults to medium, loads from chrome.storage.local, and applies live to the side panel.
  • Each accepted prompt submission snapshots the current effort and includes it in session.send as promptEffort.
  • The bridge validates promptEffort, passes it into BridgeSessionManager, includes it in AgentSendInput, and sends it to Codex App Server as effort on every production turn/start, including default medium turns.
  • Effort is never embedded in the user prompt string and never shown as assistant activity or reasoning text.
  • Existing payload-size preflight still includes the full structured session.send message, now including promptEffort.

What We're NOT Doing

  • No model selector. This spec only controls reasoning effort for the existing Codex model selection.
  • No dynamic model/list capability UI. Sidra will use the current Codex harness effort values: low, medium, high, xhigh.
  • No per-provider abstraction that pretends all providers support effort. V1 provider is Codex; the shared protocol field is provider-affecting but constrained.
  • No effort text inside formatPromptForAgent. Effort must stay structured to avoid prompt injection and transcript leakage.
  • No automatic transcript marker for effort changes. Prompt effort is a send control, not page context.
  • No keyboard shortcuts for increasing/decreasing effort in V1.

D5 Map promptEffort to Codex App Server turn/start field effort. Generated App Server bindings and Sidra's existing investigation identify effort as the per-turn field. sends_prompt_effort_to_provider_input, passes_prompt_effort_as_turn_start_effort

Performance Considerations

The new field is a short string and has no meaningful size or runtime cost. The only performance risk is accidental dynamic model discovery on every prompt; that is out of scope.

Migration Notes

  • Existing stored settings lack promptEffort; parsing must default to medium.
  • Existing extension builds that omit promptEffort should still parse as medium on the bridge side during development.
  • No persisted transcript or provider session migration is needed because Sidra V1 has no persistent chat history.
