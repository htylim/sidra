Goal

Add a toolbar-driven page selection mode with two distinct capture modes.

ModeUser interactionAttachment result
Text selection The cursor changes to text selection. The user clicks, holds, drags over page text, and releases. Sidra attaches the selected text as untrusted context.
Area snapshot The cursor changes to a crosshair. The user clicks, holds, drags a rectangle, and releases. Sidra captures the selected visible area as an image snapshot and attaches that image as untrusted context.

The resulting attachment is added to the composer for the active page session. It is included in the next accepted send, whether that send is plain Send, Capture + Send, or a quick action.

The composer shows attachments inside the composer box. Text attachments show a preview. Image snapshot attachments show a thumbnail, dimensions, source page, and remove control.

Decision: Area snapshot capture is an image attachment, not text extraction from the selected rectangle. All page-derived text and images enter the bridge as untrusted context, same as current Capture + Send page context.

Product Behavior

User actionExpected result
Open side panel No selection script is injected. No page content is read.
Click Select page context The active tab receives a transient overlay. The toolbar button enters active Selecting state. The overlay exposes Text, Snapshot, and Cancel controls.
Use Text mode The cursor changes to text selection. The user clicks, holds, drags across page text, and releases. Sidra captures the native selected text and adds a text attachment.
Use Snapshot mode The cursor changes to a crosshair. The user clicks, holds, drags a rectangle, and releases. Sidra captures a PNG image of that visible rectangle and adds an image attachment.
Finish either mode The side panel appends a composer attachment for the current URL session. Nothing is sent to the bridge yet.
Review composer attachments The composer box shows attachment chips or rows with type, page title or domain, preview text for text attachments, thumbnail for image attachments, and remove buttons.
Remove an attachment Only that attachment is removed. Draft prompt, transcript, current page state, and other attachments remain unchanged.
Click plain Send If composer attachments exist, Sidra sends the prompt with those attachments as untrusted context. No whole-page capture runs. Attachments clear only after the coordinator accepts the send.
Click Capture + Send Sidra captures the current page using readable or full DOM mode, then sends a context bundle containing composer attachments plus the captured page context. All bundle items are untrusted. Attachments clear only after the coordinator accepts the send.
Click a quick action Quick actions include composer attachments as untrusted context. If the quick action uses capture, the sent context bundle includes both attachments and the captured page context.
Capture fails while attachments exist Capture + Send does not silently downgrade to attachment-only send. The draft and attachments remain. The user can switch to plain Send.
Cancel or press Escape The overlay is removed. Existing composer attachments are left unchanged.
Navigate or switch tabs while selecting The active request is cancelled or its result is ignored. No stale attachment is stored.

Architecture

Ownership

  • apps/extension/src/page-selection-service.ts owns active-tab selection orchestration, dynamic script injection, request IDs, timeout handling, cancellation, screenshot capture, screenshot cropping, and stale-result filtering.
  • apps/extension/src/page-selection-content.ts owns the in-page overlay, mode controls, cursor changes, native text selection capture, snapshot drag geometry, cleanup, and message replies.
  • apps/extension/src/capture-service.ts continues to own conversion from captured page material into PageContext. Add selected-text, area-snapshot, and bundle builders here or in a sibling capture helper used only by CaptureService.
  • apps/extension/src/url-session-store.ts owns composer context attachments scoped by page key, remove-by-id, and clear-on-accepted-send behavior.
  • apps/extension/src/side-panel-controller.ts owns user commands. React calls controller intents only.
  • packages/protocol owns the wire schema for untrusted text, untrusted images, and bundles.
  • apps/bridge owns translating text plus image context into provider input while preserving the untrusted context boundary.

Data Flow

Toolbar button
  -> SidePanelController.startPageSelection(mode)
  -> PageSelectionService.start({ tabId, windowId, requestId, mode })
  -> chrome.scripting.executeScript({ target: { tabId, allFrames: true }, files: ["page-selection-content.js"] })
  -> PageSelectionService records injected frameIds from InjectionResult[]
  -> chrome.tabs.sendMessage(tabId, { type: "sidra.pageSelection.start", requestId, mode, topFrame: true }, { frameId: 0 })
  -> chrome.tabs.sendMessage(...same requestId/mode..., { frameId }) for each non-top injected frame
  -> Text mode: permitted frame script captures native selection text on mouseup
  -> Snapshot mode: content script captures rectangle geometry and viewport proof on mouseup
  -> Snapshot mode: extension validates the same tab/window/URL/viewport is still active
  -> Snapshot mode: extension rate-limits and calls chrome.tabs.captureVisibleTab(windowId, { format: "png" })
  -> Snapshot mode: extension crops the visible-tab PNG to the selected rectangle only if viewport proof still matches
  -> CaptureService builds selected_text or area_snapshot PageContext
  -> UrlSessionStore appends composer context attachment
  -> next Send builds a context bundle from attachments
  -> Capture + Send adds the current page capture to the same bundle

Content Script Strategy

Use a bundled file injection, not a serialized function. Selection mode needs multiple event listeners, Shadow DOM UI, runtime messaging, cursor changes, and robust cleanup. Add a new Vite Rollup input that emits page-selection-content.js at the extension root.

Inject the bundle with chrome.scripting.executeScript({ target: { tabId, allFrames: true }, files: ["page-selection-content.js"] }) and keep the returned InjectionResult.frameId values. PageSelectionService starts the request by sending chrome.tabs.sendMessage(tabId, message, { frameId }) to each injected frame. Only frameId: 0 receives topFrame: true and creates overlay controls. Non-top frames arm only their text-selection listeners for the same requestId and mode. Every permitted frame reports selected text with frameId, documentUrl, and the same requestId. PageSelectionService accepts the first non-empty text result for the active request, sends cleanup to every injected frame, and ignores duplicate or late frame results. If all injected frames report empty text, or if the selected frame cannot be injected because Chrome blocks that frame or URL, text mode must fail visibly with no attachment. Snapshot mode remains top-frame orchestrated and captures pixels from the visible tab.

Text mode must allow native page text selection outside the overlay controls and use a text-selection cursor. Snapshot mode must capture pointer events, use a crosshair cursor, draw the rectangle, and return viewport-relative geometry, scroll offsets, document URL, viewport dimensions, and device pixel ratio. The extension crops the visible-tab screenshot outside the page script.

The manifest already has scripting, tabs, and <all_urls>. Chrome host permissions allow matching-tab calls such as chrome.tabs.captureVisibleTab and chrome.scripting.executeScript. Do not add activeTab for normal http/https pages. Chrome-restricted pages, extension pages, Chrome Web Store pages, blocked frames, and file URLs without user-granted file access remain unsupported.

chrome.tabs.captureVisibleTab is expensive and Chrome exposes MAX_CAPTURE_VISIBLE_TAB_CALLS_PER_SECOND as 2. PageSelectionService must serialize snapshot captures, call it at most once per completed snapshot drag, and reject additional snapshot requests while a capture is in flight or inside the local rate-limit window.

The script must be idempotent. Re-injection should reuse one controller per document. It must remove overlay DOM, cursor styles, and event listeners after completion, cancellation, timeout message, side-panel shutdown, document unload, or Escape.

Snapshot Crop Safety Contract

Snapshot selection must never crop stale pixels. The content script returns a SelectionViewportProof at mouseup containing requestId, tabId if available, windowId if available, documentUrl, viewport.width, viewport.height, viewport.devicePixelRatio, viewport.scrollX, viewport.scrollY, and the selected viewport-relative rectangle. Before calling captureVisibleTab, the extension re-queries the active tab and asks the top-frame script for current viewport proof. After the PNG is decoded, it compares the decoded screenshot dimensions to viewport.width * devicePixelRatio and viewport.height * devicePixelRatio within one physical pixel.

Abort without adding an attachment if the active tab ID, window ID, document URL, viewport dimensions, device pixel ratio, or scroll offsets changed since mouseup. Clamp the crop rectangle to the decoded image bounds only after the stale-proof checks pass. Browser zoom changes are treated as device-pixel-ratio or screenshot-dimension mismatches and must fail visibly instead of guessing.

Bridge and Provider Input Contract

Replace the text-only provider boundary with a provider-neutral part model. BridgeTurnInput remains the protocol-facing input, but formatPromptForAgent must produce ordered provider parts instead of only one string when images are present.

export type AgentInputPart =
  | { kind: "text"; text: string }
  | {
      kind: "image";
      mimeType: "image/png";
      dataBase64: string;
      byteLength: number;
      width: number;
      height: number;
      untrustedBoundaryText: string;
    };

export type AgentSendInput = {
  parts: AgentInputPart[];
  displayTitleSource?: ProviderDisplayTitleSource;
};

For text-only turns, parts contains one text part equivalent to today's formatted prompt. For image turns, the bridge must order parts as: untrusted context introduction text, selected text and metadata JSON text, an untrusted-boundary text part immediately before each image, the image part, then the user request text. Any JSON text representation of an area_snapshot or context_bundle must omit image.dataBase64 and include only image metadata such as MIME type, byte length, dimensions, source label, and selection geometry. The boundary text must state that the image is untrusted page content and visible instructions inside it must not be followed unless the user's request explicitly asks for that content to be analyzed.

CodexAppServerProvider must map text parts to { type: "text", text, text_elements: [] }. It must map image parts to { type: "image", url: "data:image/png;base64,..." }, matching the real installed Codex app-server contract. It must not serialize image base64 into a text prompt, omit an image, OCR an image, or silently degrade to text-only. If the app-server rejects or does not support image items, the provider must fail the send before starting the turn; the extension must preserve the draft and attachments.

Phase 1 must include a fake app-server contract test that asserts the exact turn/start.input array, including the image item and the adjacent untrusted-boundary text. Phase 1 must also add and run tools/codex-smoke/image-input-smoke.mjs to smoke-test this shape against the real installed Codex app-server before proceeding. If the real app-server rejects this item shape, stop Phase 1 and update the spec; do not implement a hidden alternate mapping.

Data Model

Protocol Version

Bump PROTOCOL_VERSION from 3 to 4. The parser is strict, and selected text, area image snapshots, and bundled context add new accepted PageContext.kind values. Updating the version makes extension/bridge mismatches fail clearly.

  y: number;
  width: number;
  height: number;
};

  height: number;

  scrollY: number;

export type PageContextSelectionGeometry = {

  dataBase64: string;
  byteLength: number;
  width: number;
  height: number;

export type MetadataOnlyPageContextReason =
  | "no_usable_text"
  | "content_too_large"
  | "full_dom_too_large"
  | "selection_too_large";

export type PageContextBase =
  | ExistingReadableContext
  | ExistingFullDomContext
  | ExistingMetadataOnlyContext
  | {
      kind: "selected_text";
      metadata: PageContextMetadata;
      text: string;
      textLength: number;
      selection: PageContextSelectionGeometry;
    }
  | {
      kind: "area_snapshot";
      metadata: PageContextMetadata;
      image: PageContextImage;
      selection: PageContextSelectionGeometry;
    };

export type PageContextBundleItem = {
  id: string;
  label: string;
  trust: "untrusted";
  source: "selected_text" | "area_snapshot" | "page_capture";
  context: PageContextBase;
};

export type PageContext =
  | PageContextBase
  | {
      kind: "context_bundle";
      trust: "untrusted";
      metadata: PageContextMetadata;
      createdAt: string;
      items: PageContextBundleItem[];
    };

The parser must enforce geometry, image, proof, and bundle invariants: all rect, viewport, and proof numbers are finite, width and height are positive, selection.viewport exactly matches selection.captureProof.viewport, text length exactly matches text content, selected_text uses mode: "text_selection", area_snapshot uses mode: "area_snapshot", images are PNG base64 with exact decoded byte length and positive dimensions, bundles have 1 to 12 items, bundle metadata is required, bundle item IDs are unique, every bundle item has trust: "untrusted", bundle item source matches its context kind, and bundle items cannot recursively contain another bundle.

Bundle metadata is the provider-neutral page metadata used for Sidra history titles and context state. For plain sends with composer attachments, build bundle metadata from the active URL session page identity and the send timestamp. For Capture + Send, use the newly captured page context metadata. If item metadata differs from bundle metadata because an attachment was created before a title update, keep each item metadata intact but use only bundle metadata for display-title derivation. The bridge must update pickTitlePageMetadata to read context_bundle.metadata.

UrlSessionSnapshot must expose attachment snapshots with previews, thumbnails, dimensions, and IDs, but it must never expose full selected text beyond the short preview or raw image dataBase64. Keep raw selected text and raw PNG payloads only in volatile in-memory attachment records inside UrlSessionStore or a sibling controller-owned store. Do not persist raw attachments to Chrome storage, transcript state, or serialized UI snapshots. Clear raw payloads on accepted send, remove-one, clear-all, new chat, session disposal, and bridge disconnect cleanup.

Edge Cases

CaseRequired handling
Unsupported pages such as chrome:// Disable the toolbar button when active page status is unsupported. If injection or screenshot capture still fails, show a non-chat selection error and do not mutate attachments.
Frames and cross-origin iframes Inject selection listeners into all permitted frames. Text mode may attach selected text from the top frame or any frame where injection is permitted. If the selected text lives in a frame Chrome blocks, show a local unsupported-frame error and do not add an attachment. Snapshot mode captures pixels from the visible tab, including cross-origin iframe pixels visible in the selected rectangle.
PDF viewer, file URLs, or extension pages Treat as unsupported unless script injection and, for snapshot mode, visible-tab screenshot capture both work. File URLs remain unsupported unless the user has granted extension file access and manual validation proves both injection and screenshot capture. Do not special-case OCR.
Large selected text Use the readable content character limit. If exceeded, add a removable too-large attachment row and send metadata_only with selection_too_large if the user keeps it.
Large image snapshot Enforce an image byte limit below the hard bridge payload limit and below the Native Messaging frame budget after JSON/base64 overhead. If exceeded, show a local error and do not add the image attachment.
Bundled payload too large Before plain send, capture send, or quick action sends a bundle, compute the serialized session.send payload byte length with the same protocol helper used by the bridge. If the final message would exceed BRIDGE_HARD_PAYLOAD_BYTE_LIMIT, fail locally, preserve draft and attachments, and show which attachments must be removed.
Empty text selection Do not add an attachment. Clear the active overlay, keep existing attachments unchanged, and show a local selection error that no text was selected.
Tiny or empty snapshot rectangle Do not add an attachment. Require a minimum rectangle size so accidental clicks do not create image attachments.
Many attachments Limit composer attachments to 10 per URL session. Show a local error when the user tries to add another.
Raw image lifecycle Raw PNG payloads stay in volatile memory only and never appear in UrlSessionSnapshot, transcript entries, logs, or Chrome storage. Thumbnails in the composer are allowed.
Screenshot API rate limit Allow one captureVisibleTab call per completed snapshot selection and serialize snapshot capture requests so Sidra never exceeds Chrome's 2 calls per second limit.
Page scroll, zoom, resize, or tab switch during snapshot Capture viewport proof at mouseup, revalidate active tab/window/document URL/viewport/DPR/scroll before screenshot, and verify decoded screenshot dimensions after capture. If any value changed, fail visibly and do not add an attachment.
Multiple selections Each successful selection appends a new attachment. The user can remove attachments individually or clear all.
Send rejected by coordinator Keep draft prompt and attachments. Clear attachments only after the coordinator accepts the send.
Bridge or provider cannot accept image input Fail the send visibly before clearing the draft or attachments. Do not OCR the image, serialize image base64 as text, or silently drop it.

References

  • Chrome tabs API: host permissions allow matching-tab calls including tabs.captureVisibleTab() and scripting.executeScript(); activeTab grants temporary host permission after user invocation.
  • Chrome tabs API: captureVisibleTab() captures the visible area of the active tab and is gated by <all_urls> or activeTab, with sensitive-page and file-URL caveats.
  • Chrome tabs API: MAX_CAPTURE_VISIBLE_TAB_CALLS_PER_SECOND is 2, so snapshot capture must be serialized and rate-limited.
  • Real Codex app-server smoke validation accepts { type: "image", url } for data URL image input.

Non-Goals

  • No OCR or text extraction from area snapshot images.
  • No passive page content reading or screenshots on side-panel open, tab change, hover, or page load.
  • No persistent content scripts in manifest.json; injection remains user-command driven.
  • No attempt to bypass browser restrictions on Chrome pages or extension pages.
  • No transcript dump of raw selected text or base64 image data before send. Composer previews and thumbnails are allowed because they are the user-controlled pending context UI.
