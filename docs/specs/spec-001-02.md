# Bridge Concurrent And Cancellable Session Runtime Implementation Plan

## Overview

Refactor the bridge runtime so provider sessions and in-flight turns are managed per client session. The bridge must support one in-flight prompt per client session, concurrent prompts across different client sessions, and cancellation of an active turn.

## Current State Analysis

`apps/bridge/src/index.ts` keeps a `Map<string, AgentSession>` and handles all commands in one `switch`. A new `AbortController` is created for every `session.send`, but it is not retained, so no later command can cancel the turn. `apps/bridge/src/native-messaging.ts` chains every incoming native message onto one `messageQueue` promise, which means a long provider stream can block subsequent commands, including commands for other client sessions.

This will break V1 behavior: each URL session may have one running turn, other URL sessions can run concurrently, and Escape/Cancel must abort the running provider operation.

## Desired End State

- `BridgeSessionManager` owns all provider sessions and per-session in-flight state.
- `session.send` starts an in-flight turn only when that client session is not already running.
- Different client sessions can stream concurrently.
- `session.cancel` aborts the active turn for one client session and does not affect other sessions.
- Native Messaging frame handling reads and dispatches later frames without waiting for unrelated provider streams to finish.
- Existing issue-001 start/send mock behavior still passes.

### Key Discoveries

- `createBridge` owns sessions directly in `apps/bridge/src/index.ts:22`.
- `session.send` creates but discards the `AbortController` in `apps/bridge/src/index.ts:65`.
- Native message handling serializes all messages through `messageQueue` in `apps/bridge/src/native-messaging.ts:9`.
- The PRD requires one in-flight prompt per URL session and concurrent prompts across other URL sessions.

## What We're NOT Doing

- Not integrating Codex.
- Not implementing permission requests.
- Not implementing heartbeat cleanup.
- Not implementing New Chat reset or session close; that is covered by `spec-001-03.md`.
- Not changing the extension UI.

## Implementation Approach

Add the minimal protocol support needed for cancellation, then move bridge session behavior into a dedicated manager. The native transport should remain responsible only for framing and safe dispatch, while provider streams run independently per session.

### Overview

Add `session.cancel` as an extension-to-bridge command and add a cancellable agent event shape for bridge-to-extension events.

## Phase 2: Introduce BridgeSessionManager

### Overview

Move provider-session ownership and send orchestration out of `createBridge`.

### Why This Phase Can Be Validated Independently

### Changes Required

#### 1. Tests (RED)

**File**: `apps/bridge/src/session-manager.test.ts`

Add these tests:

```ts
describe("BridgeSessionManager", () => {
  it("creates one provider session per clientSessionId");
  it("closes an existing provider session before replacing it on session.start");
  it("emits session_not_started when sending before session.start");
  it("streams provider events with the matching clientSessionId");
  it("rejects a second prompt in the same client session while one is in flight");
  it("allows different client sessions to stream concurrently");
});
```

The concurrency test should use two provider sessions whose async iterables can be advanced independently. It must prove session B can emit before session A finishes.

#### 2. Implementation (GREEN)

**File**: `apps/bridge/src/session-manager.ts`

Create:

```ts
export type AgentSendInput = {
  prompt: string;
  pageContext?: PageContext;
};

export type AgentSessionOptions = {
  clientSessionId: string;
  providerId: ProviderId;
};

export type ManagedSession = {
  providerSession: AgentSession;
  inFlight?: {
    controller: AbortController;
    done: Promise<void>;
  };
};
```

`BridgeSessionManager` should own:

- `startSession(clientSessionId, providerId)`
- `sendPrompt(clientSessionId, input)`
- `cancelTurn(clientSessionId)`
- event emission through an injected `emit(BridgeToExtension)`.

**File**: `apps/bridge/src/index.ts`

Replace direct session map handling with manager calls. `createBridge` should parse commands and delegate.

## Phase 3: Implement Per-Session Cancellation

### Overview

Use retained abort controllers so `session.cancel` can stop an active provider stream.

### Why This Phase Can Be Validated Independently

Cancellation can be tested directly against `BridgeSessionManager` with fake providers that observe the abort signal.

### Changes Required

#### 1. Tests (RED)

**File**: `apps/bridge/src/session-manager.test.ts`

Add these tests:

```ts
describe("BridgeSessionManager cancellation", () => {
  it("aborts the active provider send for the target client session");
  it("emits assistant.cancelled for the target client session");
  it("does not cancel an in-flight turn in another client session");
  it("allows a later prompt after cancellation completes");
  it("emits session.error when cancelling a session with no in-flight turn");
});
```

#### 2. Implementation (GREEN)

**File**: `apps/bridge/src/session-manager.ts`

Implement `cancelTurn(clientSessionId)`:

- Find the managed session.
- If no session or no `inFlight`, emit `session.error` with code `no_in_flight_turn`.
- Call `AbortController.abort()`.
- Ensure the stream loop exits without emitting further assistant text for that turn.
- Emit `{ type: "agent.event", event: { type: "assistant.cancelled" } }`.
- Clear `inFlight`.

**File**: `apps/bridge/src/index.ts`

Dispatch parsed `session.cancel` to `manager.cancelTurn`.

## Phase 4: Stop Native Transport From Blocking Unrelated Streams

### Overview

Refactor Native Messaging dispatch so a long provider stream does not block later messages for unrelated sessions.

### Why This Phase Can Be Validated Independently

A native framing test can feed multiple frames and assert later-session events are emitted before an earlier long stream completes.

### Changes Required

#### 1. Tests (RED)

**File**: `apps/bridge/src/native-messaging.test.ts`

Add these tests:

```ts
describe("native messaging dispatch", () => {
  it("dispatches later session messages while an earlier provider stream is still open");
  it("can process session.cancel while a provider stream is still open");
  it("continues to report invalid JSON without crashing");
});
```

The first test should inject a custom bridge or provider that holds session A open while session B completes.

#### 2. Implementation (GREEN)

**File**: `apps/bridge/src/native-messaging.ts`

Change the dispatch model:

- Frame parsing stays sequential and synchronous.
- JSON parse errors emit immediately.
- Valid messages are handed to `bridge.handleMessage(message)` without awaiting unrelated provider streams.
- Each dispatch attaches its own `.catch(...)` handler to emit `bridge.error`.

If any ordering is required for commands targeting the same client session, enforce that inside `BridgeSessionManager`, not in the transport.

## Testing Strategy

Use fake providers with manually controlled async iterables to prove concurrency and cancellation deterministically. Keep smoke coverage for the issue-001 mock chat path.

## Performance Considerations

The bridge should no longer block every Native Messaging command behind the slowest provider stream. This is required for multiple URL sessions and cancellation responsiveness.

## Migration Notes

Existing `createBridge({ emit })` tests should continue to work through the new manager. Avoid logging prompts or page content in new errors or tests.
