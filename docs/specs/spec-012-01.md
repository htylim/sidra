# Session Shutdown And Heartbeat Cleanup Implementation Plan

## Overview

Implement issue 012: extension heartbeats and Native Messaging connection lifecycle must scope bridge provider sessions to a live side panel connection. When the side panel disconnects or heartbeats stop, bridge cleanup must cancel in-flight turns, close provider sessions, and avoid logging prompts or page content.

The work spans the extension bridge connection, side-panel session cleanup, bridge runtime connection ownership, and bridge session manager cleanup APIs. React remains a view and must not own heartbeat or cleanup sequencing.

## Current State Analysis

Issue 012 requires regular extension `heartbeat` messages, bridge session tracking by native connection, provider session cleanup on disconnect or heartbeat timeout, local in-memory extension session clearing where detectable, safe in-flight cancellation/error state, and tests for heartbeat success, heartbeat timeout, native disconnect, in-flight cleanup, and no prompt/content logging: `docs/issues/issue-012.md:13`.

Current code already has useful pieces:

- Protocol v2 already defines and validates `heartbeat`, so this issue should not add a new protocol version or message shape: `packages/protocol/src/index.ts:48`, `packages/protocol/src/index.ts:241`.
- `BridgeConnection` owns `chrome.runtime.connectNative`, port readiness, retry, disconnect, and raw bridge message validation: `apps/extension/src/bridge/connection.ts:49`.
- `SidePanelController` already marks URL sessions disconnected when the native port disconnects and before retrying the bridge: `apps/extension/src/side-panel-controller.ts:169`, `apps/extension/src/side-panel-controller.ts:273`.
- `UrlSessionStore.markBridgeDisconnected()` clears all session approvals and marks provider-backed coordinators disconnected: `apps/extension/src/url-session-store.ts:258`.
- `BridgeSessionCoordinator.markBridgeDisconnected()` clears queued submissions, pending permissions, running state, and records a safe `Bridge disconnected` status: `apps/extension/src/bridge/session-coordinator.ts:328`.
- `BridgeSessionManager` owns provider sessions, in-flight turn state, permission requests, cancellation, reset, and close for each `clientSessionId`: `apps/bridge/src/session-manager.ts:78`.
- `BridgeSessionManager.closeSession()` already aborts and closes one provider session through `closeManagedSession()`: `apps/bridge/src/session-manager.ts:177`.
- `createBridge()` currently treats `heartbeat` as a no-op and has no connection-level owner or close-all path: `apps/bridge/src/index.ts:19`, `apps/bridge/src/index.ts:71`.
- `runNativeMessagingBridge()` represents one Native Messaging connection per process invocation, emits `bridge.ready`, dispatches messages concurrently, and currently has no input-close cleanup or heartbeat timeout wiring: `apps/bridge/src/native-messaging.ts:14`, `apps/bridge/src/native-messaging.ts:37`.

Important constraints:

- Architecture assigns extension heartbeat connection state to `apps/extension/src/bridge`, provider lifecycle and heartbeat/disconnect cleanup to `apps/bridge`, and protocol shape to `packages/protocol`: `docs/ARCHITECTURE.md:13`, `docs/ARCHITECTURE.md:16`, `docs/ARCHITECTURE.md:98`.
- The side panel view must not own heartbeat cleanup, connection lifecycle, or provider lifecycle: `docs/ARCHITECTURE.md:84`.
- Different URL sessions must continue to run concurrently while one session has an in-flight turn: `docs/ARCHITECTURE.md:102`.
- Permission and provider events must not include prompts, captured page content, raw tool input, stdout, stderr, or private reasoning. Cleanup should keep that same privacy rule: `docs/specs/spec-011-01.md:113`.

## Desired End State

Heartbeat and shutdown behavior is owned by existing boundaries:

- `BridgeConnection` starts a periodic `heartbeat` loop only while its native port is connected, stops it on disconnect/retry/manual disconnect, and exposes lifecycle hooks for detectable side-panel teardown.
- `SidePanelController` preserves visible URL-session state on bridge crash/retry by marking sessions disconnected, and clears extension-owned in-memory URL sessions only when side-panel teardown is detectable.
- `UrlSessionStore` exposes a clear-all shutdown boundary that drops in-memory URL-session records and clears approvals without asking React to coordinate per-session state.
- The bridge runtime treats each `createBridge()` or Native Messaging process as one connection owner. Every session started through that bridge belongs to that connection.
- `BridgeSessionManager` exposes a close-all cleanup API that aborts in-flight turns, rejects pending permissions, closes provider sessions, clears the session map, and emits only safe cancellation/error events when appropriate.
- `createBridge()` updates heartbeat time on `heartbeat`, owns the timeout watchdog, and closes connection-owned sessions on timeout.
- `runNativeMessagingBridge()` only owns framed IO lifecycle. It calls bridge connection cleanup when stdin closes, errors, or ends.

No new protocol fields are needed. Existing `heartbeat` is enough because Native Messaging already scopes the bridge process to one extension connection.

Behavior matrix:

| Case | State | Event/action | Expected behavior | Required tests |
| --- | --- | --- | --- | --- |
| Heartbeat starts | Extension connects native port | `bridge.ready` or connected port | Extension posts periodic `heartbeat` on the live port | `starts_heartbeat_loop_when_native_port_connects` |
| Heartbeat stops on disconnect | Native port disconnects | Port `onDisconnect` | Extension stops heartbeat timer and does not post to stale port | `stops_heartbeat_loop_when_native_port_disconnects` |
| Retry replaces heartbeat | Connected port retried | `retryBridge()` | Old port heartbeat stops; new port gets heartbeats after reconnect | `retry_replaces_heartbeat_loop_with_fresh_native_port` |
| Bridge disconnect | Native port disconnects while side panel remains open | Port `onDisconnect` | Extension preserves drafts/transcripts, marks provider-backed sessions disconnected, and waits for retry | `native_disconnect_preserves_visible_url_sessions_and_marks_them_disconnected` |
| Manual shutdown | Side panel teardown detectable | `pagehide`/`beforeunload` or controller shutdown | Extension clears URL-session memory and disconnects the port | `side_panel_pagehide_disconnects_bridge_and_clears_url_sessions` |
| Bridge heartbeat accepted | Bridge receives `heartbeat` before timeout | Valid message | Timeout deadline extends and sessions remain open | `heartbeat_extends_connection_deadline_without_closing_sessions` |
| Heartbeat timeout | Started sessions, no heartbeat | Watchdog fires | Bridge aborts in-flight turns, rejects pending permissions, closes all provider sessions, and clears session map | `heartbeat_timeout_closes_all_connection_sessions` |
| Native disconnect | stdin ends/errors/closes | Native connection closes | Bridge cleanup runs once and closes all provider sessions | `native_disconnect_closes_all_connection_sessions_once` |
| Start cleanup race | Provider session creation pending | Timeout/disconnect | Newly-created provider session cannot leak after cleanup wins | `closeAll_during_session_start_closes_late_created_provider_session` |
| In-flight cleanup | Provider turn streaming or permission-blocked | Timeout/disconnect | Turn aborts, pending permission rejects, provider closes, no stale session remains reusable | `closeAll_aborts_in_flight_turns_and_rejects_pending_permissions` |
| Safe cleanup state | Cleanup cancels turn | Bridge emits or records state | Emitted messages contain safe cancellation/error status only, not prompt or page content | `cleanup_does_not_emit_prompt_or_page_content` |
| Concurrency | Multiple sessions owned by same connection | Cleanup event | All sessions for that connection close; no unrelated process/connection is affected | `closeAll_closes_only_sessions_owned_by_this_manager` |

## What We're NOT Doing

- No protocol version bump or new heartbeat payload. `heartbeat` already exists.
- No persistent session storage or reconnect restoration.
- No background-page global heartbeat. The side panel connection owns this lifecycle.
- No React-owned heartbeat timer or provider cleanup.
- No Codex SDK integration. Issue 013 owns the real provider adapter.
- No prompt, captured page content, raw provider input, stdout, stderr, or private reasoning in cleanup logs, errors, or tests.

## Implementation Approach

Use three TDD phases:

1. Add extension heartbeat and local shutdown hooks at the bridge connection/controller boundary.
2. Add bridge-side connection cleanup and heartbeat timeout ownership.
3. Add Native Messaging disconnect integration and final privacy/concurrency coverage.

Keep timing configurable in tests. Use fake timers for heartbeat intervals and timeout watchdogs. Keep production defaults conservative and named, for example a heartbeat interval around 10 seconds and a timeout around 30 seconds, but make exact values constants owned by the bridge/connection modules.

Extension unit tests can prove that a live native port gets heartbeats, stale ports do not, retry replaces timers, and controller shutdown clears local session state without bridge runtime changes.

#### 2. Implementation (GREEN)

**File**: `apps/bridge/src/session-manager.ts`

Changes:

- Add `closeAllSessions(reason)` or `cleanupConnection(reason)`.
- Snapshot current managed sessions, mark the manager as closing or advance a connection generation, clear the session map first, abort in-flight controllers, reject pending permissions, await each `done`, and close each provider session.
- Make cleanup idempotent and safe if called during start/send/reset/close operations.
- Guard `startSession()`/`replaceSession()` so a provider session that resolves after connection cleanup is immediately closed and never inserted into the active session map.
- Emit only safe generic cancellation/error messages if the implementation chooses to notify the extension. Do not include prompt, page context, provider exception details, raw tool input, stdout, stderr, or private reasoning.
- Preserve per-session operation serialization and unrelated-manager isolation.

**File**: `apps/bridge/src/index.ts`

Changes:

- Treat `createBridge()` as the owner for one Native Messaging connection.
- Add heartbeat state and configurable timeout options, for example `heartbeatTimeoutMs`, `setTimeout`, and `clearTimeout` for tests.
- On every valid `heartbeat`, update the deadline and reschedule timeout cleanup.
- Expose a `closeConnection(reason)` or `shutdown(reason)` method that calls `BridgeSessionManager.closeAllSessions(...)` and stops the timeout.
- Keep the raw message switch as routing only. Do not put provider cleanup logic in the switch.

## Performance Considerations

Heartbeat traffic is one small protocol message per interval while the side panel is connected. Cleanup is O(number of sessions for one Native Messaging connection). Do not serialize unrelated connection/process cleanup through shared global state.
