# Sidra

Sidra is a planned Chromium side panel extension for chatting with a local AI agent about the current page.

V1 targets article and text-heavy pages on macOS. The extension will capture readable page content only when the user explicitly chooses **Capture + Send**, then send the prompt and page context to a local Native Messaging bridge that talks to Codex through `codex-sdk`.

## Development

```sh
pnpm install
pnpm --filter @sidra/extension exec playwright install chromium
pnpm test
pnpm check
pnpm build
pnpm test:e2e
```

The initial bridge smoke test verifies the prompt-to-mock-response path without Codex.

`pnpm test:e2e` currently runs the extension Playwright smoke test. It builds
`apps/extension/dist`, launches a temporary Chromium profile with that unpacked
extension loaded, verifies the Manifest V3 service worker starts, and opens the
side panel page to confirm Sidra shows the bridge setup state when the Native
Messaging host is not installed.

## Run Locally

The current runnable path is a developer install. It can load the extension and
connect it to the local Native Messaging bridge, but the bridge still uses the
mock provider. The real Codex provider and installer script are planned work.

1. Clone and install dependencies:

```sh
git clone <repo-url> sidra
cd sidra
pnpm install
```

2. Build the extension and bridge:

```sh
pnpm build
```

3. Load the unpacked extension in a Chromium browser:

- Open `chrome://extensions`.
- Enable **Developer mode**.
- Choose **Load unpacked**.
- Select `apps/extension/dist`.
- Copy the generated extension ID from the loaded Sidra card.

The extension ID is needed by the Native Messaging host manifest. Because Sidra
does not yet pin a development extension key, this ID is generated by the
browser. Loading from the same local `dist` path should keep it stable on that
machine, but another clone/path can produce a different ID.

4. Create a local wrapper for the bridge executable:

```sh
mkdir -p .local
cat > .local/sidra-agent-bridge <<'SH'
#!/bin/sh
exec node "__REPO_ROOT__/apps/bridge/dist/cli.js"
SH
perl -0pi -e "s#__REPO_ROOT__#$(pwd)#g" .local/sidra-agent-bridge
chmod +x .local/sidra-agent-bridge
```

5. Install a Native Messaging host manifest for the browser you are using.

Replace `<EXTENSION_ID>` with the ID copied from `chrome://extensions`.

For Google Chrome on macOS:

```sh
HOST_DIR="$HOME/Library/Application Support/Google/Chrome/NativeMessagingHosts"
mkdir -p "$HOST_DIR"
cat > "$HOST_DIR/com.sidra.agent_bridge.json" <<JSON
{
  "name": "com.sidra.agent_bridge",
  "description": "Sidra local agent bridge",
  "path": "$(pwd)/.local/sidra-agent-bridge",
  "type": "stdio",
  "allowed_origins": ["chrome-extension://<EXTENSION_ID>/"]
}
JSON
```

For Google Chrome for Testing on macOS, use:

```text
~/Library/Application Support/Google/ChromeForTesting/NativeMessagingHosts/com.sidra.agent_bridge.json
```

For Chromium on macOS, use:

```text
~/Library/Application Support/Chromium/NativeMessagingHosts/com.sidra.agent_bridge.json
```

Brave commonly uses:

```text
~/Library/Application Support/BraveSoftware/Brave-Browser/NativeMessagingHosts/com.sidra.agent_bridge.json
```

Helium support is not claimed by this repo yet. Issue 014 tracks verifying the
Helium manifest location, side panel compatibility, and a real installer.

6. Reload Sidra from `chrome://extensions`, open its side panel, and submit a
prompt. With the current mock bridge, the expected response is:

```text
Mock response to: <your prompt>
```

If the bridge does not connect, check:

- the side panel shows `Sidra cannot connect to the local bridge.` and a
  **Retry** button while the host is missing or unavailable;
- after fixing the host manifest or rebuilding the bridge, click **Retry** in
  the side panel instead of reloading the side panel;
- the extension has the `nativeMessaging` permission in `apps/extension/dist/manifest.json`;
- the host manifest filename is exactly `com.sidra.agent_bridge.json`;
- the host manifest `name` is exactly `com.sidra.agent_bridge`;
- `allowed_origins` contains the exact extension ID and trailing slash;
- `path` is absolute and points to an executable file;
- `pnpm build` has produced `apps/bridge/dist/cli.js`;
- the browser was restarted or the extension was reloaded after installing the host manifest.

Chrome's Native Messaging documentation defines the manifest shape, absolute
path requirement, `allowed_origins`, and the macOS host manifest directories:
https://developer.chrome.com/docs/extensions/develop/concepts/native-messaging

## V1 goals

- Provide a Chromium side panel chat UI.
- Capture page context explicitly, not passively.
- Use a local bridge process to communicate with Codex.
- Keep the extension independent from provider implementation details.
- Support multiple URL-scoped sessions while the side panel is alive.
- Surface agent activity, permission requests, errors, and cancellation clearly.
