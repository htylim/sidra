# Provider Permission Requests Implementation Plan

## Overview

Implement issue 011: provider permission requests must pause the related provider turn, render inline transcript permission cards, and resume or fail only after the user chooses Allow once, Allow for this session, or Deny.

The work spans the protocol package, bridge session manager, extension session boundary, URL-session approval policy, controller, and transcript UI. React must stay a view. It renders permission cards and calls controller commands; it must not construct protocol messages or own provider-turn sequencing.

## Current State Analysis

Issue 011 requires permission request and response protocol, inline permission cards, blocked-turn behavior, session-scoped approvals, and tests for allow once, allow session, deny, blocked-turn behavior, cross-session isolation, and approval clearing: `docs/issues/issue-011.md:13`.

Current code already provides the session architecture this should extend:

- Protocol v2 supports `session.start`, `session.send`, `session.cancel`, `session.reset`, `session.close`, `heartbeat`, assistant events, and session errors, but it has no permission request or response messages: `packages/protocol/src/index.ts:48`.
- Protocol parsers validate message shapes at the Native Messaging boundary and reject unknown commands/messages: `packages/protocol/src/index.ts:128`, `packages/protocol/src/index.ts:202`.
- `BridgeSessionManager` owns provider sessions, in-flight turn state, cancellation, reset, and close for each `clientSessionId`: `apps/bridge/src/session-manager.ts:39`.
- `BridgeSessionManager.sendPrompt()` currently awaits the provider stream until terminal event or error; there is no provider permission callback or pending permission state: `apps/bridge/src/session-manager.ts:57`.
- The bridge dispatch switch routes validated protocol commands to `BridgeSessionManager`, so any `permission.respond` command belongs there, not in raw Native Messaging code: `apps/bridge/src/index.ts:47`.
- `BridgeSessionCoordinator` owns extension-side provider-session sequencing, transcript state, running state, cancellation, reset, and bridge-message handling for one `clientSessionId`: `apps/extension/src/bridge/session-coordinator.ts:70`.
- `UrlSessionStore` owns URL session records and already has an in-memory `SessionApprovalState` with grant, has, list, and clear methods. `newChat()` clears approvals for the active session only: `apps/extension/src/url-session-store.ts:45`, `apps/extension/src/url-session-store.ts:211`, `apps/extension/src/url-session-store.ts:226`.
- `SidePanelController` exposes product commands and derived snapshots to the view, including `cancelTurn()` and `newChat()`, but no permission response command yet: `apps/extension/src/side-panel-controller.ts:54`.
- `TranscriptEntry` supports user messages, assistant turns, and status entries, but no permission card entry: `apps/extension/src/transcript.ts:5`.
- `TranscriptView` renders entries by discriminated union and can add a new permission card view without changing controller ownership: `apps/extension/src/transcript-view.tsx:21`.
- `SidePanelView` renders transcript snapshots and command callbacks; prompt send/cancel controls already derive from session snapshot state: `apps/extension/src/side-panel-view.tsx:6`.

Important constraints:

- `docs/ARCHITECTURE.md` says protocol shapes belong in `packages/protocol`, bridge provider lifecycle belongs in `BridgeSessionManager`, URL session state and approvals belong in `UrlSessionStore`, and the side panel must call application-level commands instead of owning protocol sequencing: `docs/ARCHITECTURE.md:83`.
- One URL session may have one in-flight prompt, while other URL sessions can run concurrently: `docs/ARCHITECTURE.md:102`.
- Issue 013 later integrates the real Codex provider and documents current Codex SDK APIs. This issue should add a provider-facing permission abstraction and fake-provider tests, not hard-code Codex SDK-specific permission shapes: `docs/issues/issue-013.md:13`.

## Desired End State

Provider permission handling is owned by the existing boundaries:

- `packages/protocol` defines and validates permission request/response messages.
- `BridgeSessionManager` owns pending bridge-side permission decisions for the active provider turn and resolves or rejects the provider's blocked operation.
- The provider interface exposes a small permission callback that provider adapters can call while streaming a turn.
- `BridgeSessionCoordinator` turns permission request messages into transcript entries, tracks pending/resolved permission cards for its `clientSessionId`, posts permission responses through the transport, and keeps the turn in flight until the provider emits a terminal event or a terminal error.
- `UrlSessionStore` owns session-scoped approval policy for URL sessions and clears approvals on New Chat. It must also expose a close/shutdown clearing path for all sessions when issue 012 wires shutdown cleanup.
- `SidePanelController` exposes `respondToPermission(requestId, decision)` as an application command. It routes the response to the active URL session and records session approvals only for Allow for this session.
- `SidePanelView` and `TranscriptView` render permission cards and call the controller command. They do not inspect Native Messaging details.

Protocol shape:

```ts
export type PermissionRequest = {
  requestId: string;
  permissionKey: string;
  title: string;
  description?: string;
  metadata?: {
    toolName?: string;
    commandPreview?: string;
  };
};

export type PermissionDecision = "allow_once" | "allow_for_session" | "deny";

export type BridgeToExtension =
  | ...
  | {
      type: "permission.request";
      version: 2;
      clientSessionId: string;
      request: PermissionRequest;
    };

export type ExtensionToBridge =
  | ...
  | {
      type: "permission.respond";
      version: 2;
      clientSessionId: string;
      requestId: string;
      decision: PermissionDecision;
    };
```

The schema must stay intentionally small and safe. It may expose a title, description, tool name, and command preview, but it must reject prompt text, captured page content, raw tool input objects, chain-of-thought, stdout, stderr, or arbitrary metadata.

Behavior matrix:

| Case | State | Event/action | Expected behavior | Required tests |
| --- | --- | --- | --- | --- |
| Protocol accepts request | Bridge emits valid request | Parse bridge-to-extension | Request validates with `clientSessionId`, `requestId`, `permissionKey`, and safe display fields | `accepts_permission_request_message` |
| Protocol accepts response | Extension responds | Parse extension-to-bridge | Response validates only known decisions | `accepts_permission_respond_messages` |
| Protocol privacy | Request contains private/raw fields | Parse bridge-to-extension | Parser rejects the message | `rejects_permission_request_with_private_or_unknown_fields` |
| Bridge request emission | Provider asks permission mid-turn | Provider callback | Bridge emits one `permission.request` for that `clientSessionId` | `emits_permission_request_for_provider_permission_callback` |
| Blocked bridge turn | Permission pending | Provider awaits callback | No later provider events emit until a response resolves the pending request | `blocks_provider_turn_until_permission_response_arrives` |
| Allow once | Pending request | `permission.respond: allow_once` | Pending provider callback resolves as allowed; approval is not persisted | `allow_once_resumes_blocked_provider_turn_without_persisting_approval` |
| Allow for session | Pending request | `permission.respond: allow_for_session` | Pending provider callback resolves as allowed; extension records active URL-session approval | `allow_for_session_resumes_turn_and_records_active_session_approval` |
| Deny | Pending request | `permission.respond: deny` | Pending provider callback resolves as denied; provider can emit failure/terminal outcome | `deny_resumes_provider_with_denied_decision` |
| Unknown/stale response | No matching pending request | `permission.respond` | Bridge emits session error and does not affect the turn | `rejects_permission_response_without_matching_pending_request` |
| Duplicate response | Request already resolved | Second response | Bridge rejects or ignores safely; extension card remains resolved | `ignores_duplicate_permission_response_after_card_resolves` |
| Same-session second prompt | Permission pending | User sends another prompt | Send remains blocked because turn is still in flight | `pending_permission_keeps_turn_in_flight_and_blocks_duplicate_send` |
| Other session concurrency | Session A pending permission | Session B sends prompt | Session B can start and stream independently | `pending_permission_does_not_block_other_url_session` |
| Inactive session request | Session A inactive | Request arrives for A | Card is stored in A transcript; active session B does not show it | `stores_permission_request_in_matching_inactive_session_only` |
| Stale request after reset | Same `clientSessionId`, old provider turn reset | Delayed old `permission.request` arrives | Extension ignores it and does not show a fresh permission card | `ignores_permission_request_when_no_turn_is_in_flight`, `newChat_ignores_stale_permission_request_from_old_turn` |
| Active response routing | User views session A card | Click Allow/Deny | Controller posts response for A `clientSessionId` only | `respondToPermission_posts_response_for_active_session_request` |
| Cross-session isolation | Session B is active | Try responding to A request id | Controller returns false and posts no response | `respondToPermission_does_not_cross_url_session_boundary` |
| Existing session approval | Active URL session has same `permissionKey` allowed | Matching request arrives | Extension auto-responds allow-for-session for that session only and marks card allowed | `auto_allows_matching_request_from_active_session_approval` |
| Approval clearing New Chat | Active session has approval | New Chat | Active approvals clear; inactive approvals remain | Existing `newChat_clears_active_session_scoped_approvals_only` plus permission response coverage |
| Approval clearing reset/close/shutdown | Session reset/close/shutdown clears state | Lifecycle event | Session approvals and pending cards clear or resolve safely | `reset_clears_pending_permission_and_session_approvals`, `close_or_shutdown_clears_all_session_approvals` |
| Cancellation while pending | Turn pending permission | Cancel | Provider turn aborts; pending request resolves/clears and card is no longer actionable | `cancel_clears_pending_permission_request_for_turn` |

## What We're NOT Doing

- No Codex SDK integration or Codex-specific permission mapping. Issue 013 owns validating and wiring the real provider adapter.
- No persistent approval storage. Approvals are in-memory and URL-session scoped.
- No global permission allowlist UI.
- No raw tool input, prompt, page content, stdout, stderr, or private reasoning in protocol messages, logs, transcript cards, or tests.
- No React-owned protocol construction.
- No protocol generation/session id beyond the existing `clientSessionId` unless implementation proves stale same-session permission responses cannot be handled safely.

## Implementation Approach

Build this in three TDD phases:

1. Add protocol and bridge permission blocking first. This proves the provider can ask and wait, and the bridge can resolve the pending decision.
2. Add extension session state and URL-session approval policy. This proves requests and responses route to the correct URL session and approvals stay scoped.
3. Add transcript UI rendering and controller/view behavior. This proves the user can respond from inline cards without moving protocol policy into React.

If a phase reveals that approval policy needs access to inactive session records, keep that logic in `UrlSessionStore` or a small extension-owned policy collaborator. Do not move it into `SidePanelView`.

Manual checks:

1. Use a fake provider path that emits a permission request during a turn.
2. Confirm the active transcript shows one inline permission card and prompt controls stay blocked.
3. Click Allow once and confirm the turn resumes without persisting approval.
4. Repeat with Allow for this session and confirm a later same-session request with the same `permissionKey` is auto-allowed.
5. Open another URL session and confirm the same `permissionKey` is not auto-allowed there.
6. Click New Chat and confirm active-session approvals are gone.

## Performance Considerations

Permission state is O(1) per in-flight turn and small per URL session. Do not scan every transcript entry on every bridge event if a local `pendingPermissionRequestId` set can answer routing directly.

## Migration Notes

No persisted data migration is needed. Permission requests and session approvals are in-memory. Protocol stays version 2 unless implementation discovers a real compatibility need to bump it.
