# macOS Native Messaging Host Install Implementation Plan

## Overview

Implement issue 014: provide a deterministic manual developer install flow for Sidra's macOS Native Messaging host manifests for Chrome and Brave, verify Helium before claiming support, and update developer docs so a fresh local setup can load the unpacked extension, install the native host, and smoke the Codex App Server-backed bridge path.

This is a developer-install and verification issue. It should not change runtime ownership boundaries or move bridge sequencing into the extension UI.

## Current State Analysis

Issue 014 requires pinned extension ID strategy, Chrome and Brave host manifests, Helium verification before support is claimed, developer docs, and a smoke path that opens/focuses the side panel and connects to the installed bridge without bypassing the Codex App Server provider path from issue 013: `docs/issues/issue-014.md`.

Current code and docs:

- The extension already declares Manifest V3, `sidePanel`, `nativeMessaging`, and a default side panel page, but it does not include a `key`, so unpacked extension ID stability still depends on browser/path behavior: `apps/extension/public/manifest.json`.
- Manifest tests assert MV3 side panel and Native Messaging permissions, but not a pinned key or expected extension ID: `apps/extension/src/manifest.test.ts`.
- The background service worker already sets `chrome.sidePanel.setPanelBehavior({ openPanelOnActionClick: true })`, so toolbar action behavior is present and needs browser-level smoke verification, not a new UI owner: `apps/extension/public/background.js`.
- The extension's Native Messaging host name is centralized as `com.sidra.agent_bridge`: `apps/extension/src/bridge/connection.ts`.
- `BridgeConnection` remains the only extension boundary that opens `chrome.runtime.connectNative`; installer work must not leak raw Native Messaging state into React: `apps/extension/src/bridge/connection.ts`.
- Bridge production composition is gated by explicit `SIDRA_CODEX_WORKSPACE_ROOT`; without it, no provider is configured. When configured, the bridge starts `codex app-server` and checks Codex account state before exposing the provider: `apps/bridge/src/bridge-runtime.ts`.
- The bridge strips `SIDRA_CODEX_WORKSPACE_ROOT` before spawning Codex App Server, so it controls Sidra's thread working directory without leaking the control env var to Codex: `apps/bridge/src/codex-app-server-process.ts`.
- The manual E2E runbook currently documents a known-good Brave path, a hard-coded Brave extension ID for the current unpacked path, and manual manifest creation. It also still expects mock-provider text in its smoke flow, which is stale after issue 013's Codex App Server path: `docs/MANUAL-E2E-RUNBOOK.md`.
- The README still describes `codex-sdk`, mock provider behavior, unpinned IDs, and ad hoc browser manifest paths. It says Helium support is not claimed yet: `README.md`.
- The PRD expects a macOS installer script for Chrome, Brave, and Helium, but it explicitly lists Helium Native Messaging path and `sidePanel` support as open research: `docs/prd-v1.md`.

External constraints checked on 2026-05-24:

- Chrome's Native Messaging docs require a JSON manifest with `name`, `path`, `type: "stdio"`, and exact `allowed_origins`; on macOS the host `path` must be absolute, and user-level Chrome manifests live under `~/Library/Application Support/Google/Chrome/NativeMessagingHosts/`: <https://developer.chrome.com/docs/extensions/develop/concepts/native-messaging>.
- Chrome's manifest `key` field is the supported way to keep a consistent extension ID for unpacked development builds: <https://developer.chrome.com/docs/extensions/reference/manifest/key>.
- Helium's public site says Helium is Chromium-based and supports Chromium extensions, but that does not prove Sidra's `sidePanel` surface or Native Messaging host lookup path. Helium support must remain unclaimed until a local smoke verifies it: <https://helium.surf/>.
- Helium's main repo and macOS packaging repo are the primary sources for Helium-specific research: <https://github.com/imputnet/helium> and <https://github.com/imputnet/helium-macos>. The macOS packaging repo builds `Helium.app` and signs it with bundle identifier `net.imput.helium`, but this repository evidence still does not prove the browser's macOS Native Messaging manifest lookup directory or `sidePanel` behavior.
- Local Helium smoke on 2026-05-24 verified `/Applications/Helium.app` version `0.12.4.1`, bundle identifier `net.imput.helium`, Chromium `148.0.7778.178`, unpacked Sidra extension ID `dkogbjooeahemlnkpdnhhjbhojhdlnpd`, `chrome.sidePanel` API availability, `chrome.sidePanel.open({ tabId })` from a user gesture, and Native Messaging lookup from `~/Library/Application Support/net.imput.helium/NativeMessagingHosts/com.sidra.agent_bridge.json` by observing `{"type":"bridge.ready","version":2}` from `chrome.runtime.connectNative("com.sidra.agent_bridge")`. Playwright did not directly click the browser toolbar action, so toolbar `openPanelOnActionClick` still needs a human or browser-chrome-driven confirmation before claiming the exact toolbar flow.

## Desired End State

- `apps/extension/public/manifest.json` contains a pinned development `key`, and tests assert the expected development extension ID used by host manifests.
- A repo-owned macOS installer script writes deterministic Native Messaging host manifests for Chrome and Brave from explicit inputs: repo root, built bridge path or wrapper path, browser target, and pinned extension ID.
- The installer can generate and validate manifest JSON without writing to the user's browser profile, so unit tests cover manifest content and browser path selection.
- The installer creates or updates a repo-local native-host wrapper that exports the explicit `SIDRA_CODEX_WORKSPACE_ROOT` and a deterministic `PATH` entry for the `codex` executable before launching `apps/bridge/dist/cli.js`. Browser-launched native hosts must not rely on the user's interactive shell environment.
- Codex App Server startup/auth failures are surfaced to the extension as a safe `bridge.error` frame instead of only causing native-host process exit before the extension can render the setup reason. A provider setup failure must not emit `bridge.ready`, because `bridge.ready` means the side panel can use the chat path.
- The installer refuses to claim Helium by default. It can include an explicit research-only Helium write mode so verification can happen, but that mode must label the result unsupported until local checks prove the app exists, `sidePanel` works, and the manifest path is correct for the installed Helium build.
- Developer docs explain install dependencies, build commands, host installation, loading the unpacked extension, tests, and bridge troubleshooting. Docs must replace stale `codex-sdk` and mock-provider setup text with the Codex App Server path from issue 013.
- Manual smoke verifies the real developer path: toolbar action opens/focuses Sidra's side panel, the side panel connects to the installed native bridge, and provider setup/auth failures surface without falling back to the old mock provider path.

## What We're NOT Doing

- No production packaged installer, LaunchAgent, notarized app, or system-wide install.
- No Windows, Linux, Edge, Chromium, Chrome for Testing, or Firefox installer support.
- No persistent Codex credential management. Sidra uses the user's existing Codex CLI/App Server auth.
- No fallback to the mock provider for manual smoke.
- No React-owned Native Messaging setup logic.
- No Helium support claim based only on Chromium compatibility statements.

## Implementation Approach

Use three phases.

1. Pin the development extension ID and make installer manifest generation testable without touching user profile directories.
2. Wire Chrome and Brave macOS manifest installation and update current developer docs.
3. Add manual smoke verification for toolbar side-panel behavior, bridge connection, Codex App Server setup/auth failures, and Helium research gating.

Each code phase starts with tests. Documentation and manual verification phases state when TDD does not apply and replace it with concrete smoke checks.

**File**: `apps/bridge/src/bridge-runtime.ts`

Changes:

- Keep provider setup owned by the bridge runtime.
- Move initial bridge readiness emission out of transport-owned startup behavior and into the bridge runtime/protocol startup path.
- Treat missing `SIDRA_CODEX_WORKSPACE_ROOT` as a setup failure for production runtime composition, not as a providerless ready bridge. Emit a safe `bridge.error`, do not emit `bridge.ready`, keep the connection alive, and reject later chat commands with the same setup error instead of `provider_unavailable`.
- Do not create a normal `BridgeSessionManager` without a provider in the setup-failed runtime path. Own the blocking state in the bridge runtime/protocol startup layer and route every later extension command through a small setup-blocked protocol handler that validates the incoming message enough to stay safe, then emits `bridge.error` with a stable setup failure code such as `codex_setup_failed`.
- Preserve the existing `createBridge`/`BridgeSessionManager` behavior for test-only direct callers that intentionally construct a providerless bridge; this issue changes production runtime composition.
- If Codex App Server startup or auth check fails, keep the Native Messaging connection alive and emit a safe `bridge.error` frame that says Codex setup/auth failed.
- Do not emit `bridge.ready` after provider setup failure. `bridge.ready` is the extension-side readiness signal and currently enables chat.
- After provider setup failure, reject later chat commands with the same safe setup error or keep the bridge in a blocking error state. Do not let the UI progress to a providerless session that later fails as generic `provider_unavailable`.
- After provider setup succeeds, emit `bridge.ready` from the runtime/protocol startup path so `BridgeConnection` can mark the bridge ready on the existing `bridge.ready` protocol message.
- Do not include prompts, captured page content, raw App Server stderr, access tokens, or full environment values in the error frame.
- Keep the native connection alive so the side panel can render the setup failure and expose Retry after the user fixes auth or wrapper configuration.

## Testing Strategy

- Unit tests cover the pinned extension ID and pure manifest generation.
- CLI tests cover dry-run and write behavior through an injected manifest root or fake filesystem. Automated tests must not write user browser profile files.
- Existing extension tests protect runtime ownership: `BridgeConnection` remains the only extension Native Messaging port owner, and React stays a view.
- Manual smoke covers browser behavior that unit tests cannot prove: toolbar action side-panel behavior, browser manifest lookup, Native Messaging connection, and Codex App Server setup/auth surface.

## Performance Considerations

The installer writes small JSON files and does not affect runtime performance. Manual smoke should avoid long-running browser profiles and clean up temporary profiles and manifests created during verification.

## Migration Notes

Existing developers with old hand-written manifests must reinstall manifests after the pinned extension key lands because the allowed origin changes to the pinned development extension ID. Docs should tell them to remove stale manifests or rerun the installer for Chrome and Brave.
