Feasibility

This change is possible for Sidra's current supported browser set. The base behavior uses tab-specific chrome.sidePanel.setOptions({ tabId, ... }). Sidra configures tab-specific side panel availability, then lets Chromium's native toolbar side panel toggle own the per-tab open and close lifecycle.

Requirement API support Conclusion
Hide Sidra when switching to a tab where Sidra was not opened. Chrome documents tab-specific side panel options and automatic restore for tabs where the panel was previously open. Possible.
Show Sidra again when returning to a tab where it was opened. Chrome documents automatic restore when returning to a tab where the panel was previously open. Possible.
Turn Sidra off for only the current tab. Chrome owns the toolbar side panel toggle once the current tab has tab-specific options. Possible on current target browsers.
Hard boundary: Sidra does not try to synthesize browser side panel open state. The manifest targets Chromium 114+ because Sidra only relies on the base Side Panel API. Current project runbook already lists Chrome 148, Brave, and Helium Chromium 148 as known-good local targets.

Overview

Sidra currently behaves as a global side panel. Once visible, it remains visible as the user moves across tabs. This plan changes only the visibility/open state. Chat sessions, URL session state, capture behavior, bridge state, transcript state, provider sessions, and page identity behavior remain unchanged.

The new owner is a background-side SidePanelTabVisibilityController. It enables Sidra's side panel for active browser tabs and lets Chromium keep the actual per-tab open state. URL state stays in UrlSessionStore; it does not own side panel visibility.

Current State

  • apps/extension/public/background.js:1 only sets openPanelOnActionClick: true.
  • apps/extension/public/manifest.json:9 declares a global side_panel.default_path.
  • apps/extension/public/manifest.json:16 declares the toolbar action that users click to open Sidra.
  • apps/extension/src/active-page.ts:31 tracks active tab ID only to refresh page identity, not visibility state.
  • apps/extension/src/side-panel-controller.ts:109 owns side panel application commands and URL-session selection, not browser chrome visibility.
  • apps/extension/src/settings-store.ts:1 stores settings in one SidraSettings object under sidra.settings.v1.
  • apps/extension/src/options-page-view.tsx:243 already renders checkbox-style settings toggles.
  • docs/ARCHITECTURE.md:9 says apps/extension owns side panel rendering, active page tracking, settings, and browser-side app state.

Desired State

  • Opening Sidra on tab A enables and opens Sidra only for tab A.
  • Navigation inside tab A keeps Sidra visible because the tab ID does not change.
  • Switching to tab B hides Sidra unless tab B was opened earlier.
  • Returning to tab A shows Sidra again while tab A remains open.
  • Closing Sidra on tab A closes only tab A. Tab B keeps its own state.
  • Tabs created through Cmd+T, browser new-tab UI, extension pages, or link-opened child tabs start closed until Sidra is opened in that tab.

What We Are Not Doing

  • No change to URL session scoping, transcript storage, draft prompt behavior, or provider session mapping.
  • No content capture on tab switch, tab creation, side panel open, or passive background activity.
  • No URL-specific visibility. Visibility is keyed by browser tab ID only.
  • No bridge protocol changes.
  • No opener-tab inheritance. Native browser open-link-in-new-tab actions do not provide the extension user activation required by chrome.sidePanel.open.
  • No persistence of side panel open state across browser restarts.
  • No custom mirror of Chromium's internal side panel open state.

Implementation Approach

  1. Remove the global default side panel path from the manifest.
  2. Build a TypeScript background entry point so browser chrome behavior is testable.
  3. Add a SidePanelTabVisibilityController behind a small Chrome gateway.
  4. Use chrome.sidePanel.setPanelBehavior({ openPanelOnActionClick: true }) so Chromium owns toolbar open and close.
  5. Use chrome.sidePanel.setOptions({ tabId, path, enabled: true }) for the current active tab and each later activated or replaced tab.
Recommended state model 
type SidePanelAvailability = "enabled-for-tab";

Chromium owns whether the tab-specific side panel is currently open. Sidra only ensures the active tab has Sidra's side panel path available before the user clicks the toolbar action.

Risks

Risk Mitigation
Service worker suspension loses in-memory tab state. Do not mirror open state in memory. Let Chromium own the native tab-specific side panel state.
Old global side panel behavior survives through manifest default path. Remove global side_panel.default_path; use tab-specific options plus native toolbar toggle.
Action-click handling races with Chromium user activation. Do not call chrome.sidePanel.open() from the controller. Use Chromium's native toolbar side panel toggle.
New TypeScript background entry emits an unstable filename. Configure Rollup entry output so the service worker remains background.js.
