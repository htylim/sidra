# Render Streaming Assistant Turns Implementation Plan

## Overview

Implement issue 008: streamed assistant turns should render as rich transcript entries with sanitized Markdown, safe collapsed activity, escaped user text, copyable code blocks, safe links, inline errors, and URL-session isolation.

This is mostly extension-owned transcript and rendering work. The protocol must grow just enough to carry activity events whose display shape is safe by construction. The bridge/provider adapter remains responsible for never mapping private reasoning or raw provider logs into those display fields.

## Current State Analysis

Issue 008 asks for streamed assistant responses, sanitized Markdown, escaped user text, collapsed safe activity, inline operational errors, and tests for all of those behaviors: `docs/issues/issue-008.md:13`, `docs/issues/issue-008.md:17`, `docs/issues/issue-008.md:18`, `docs/issues/issue-008.md:19`, `docs/issues/issue-008.md:20`, `docs/issues/issue-008.md:21`, `docs/issues/issue-008.md:22`.

Current implementation already streams assistant text into the extension, but the transcript model is too small for V1 rendering:

- `TranscriptEntry` is only `{ role, text }`, so it cannot distinguish assistant Markdown, activity groups, error status, cancelled status, or copyable code blocks: `apps/extension/src/transcript.ts:1`.
- Assistant deltas append to the last assistant entry with plain text only: `apps/extension/src/transcript.ts:19`.
- `SidePanelView` renders every transcript entry as raw React text inside one generic `div`, so Markdown is not rendered and code/link behavior cannot exist: `apps/extension/src/side-panel-view.tsx:93`.
- The view currently relies on React text escaping for user prompts, which is good, but this should become an explicit renderer behavior covered by tests: `apps/extension/src/side-panel-view.tsx:95`.
- The protocol only has `assistant.text.delta`, `assistant.done`, and `assistant.cancelled`; it has no safe activity shape: `packages/protocol/src/index.ts:67`.
- The bridge currently forwards provider `AgentEvent` values unchanged after TypeScript typing, so issue 008 must add a runtime guard before provider events leave the bridge process.
- `BridgeSessionCoordinator` already routes `agent.event` by `clientSessionId`, ignores events for other sessions, and appends assistant text deltas in non-React code.
- `UrlSessionStore` already keeps inactive session updates out of the active snapshot and restores them when the page becomes active again.

Product constraints from the PRD:

- Assistant responses render as sanitized Markdown. Links open in a new tab. Code blocks include copy buttons. User prompts are escaped: `docs/prd-v1.md:190`.
- Internal activity is collapsed by default and may show only safe activity. Raw private chain-of-thought must not be shown: `docs/prd-v1.md:499`.
- Operational errors appear inline in the relevant chat transcript and are visually distinct: `docs/prd-v1.md:533`.

## Desired End State

The transcript model has named variants:

```ts
export type TranscriptEntry =
  | { id?: string; kind: "user_message"; role: "user"; text: string }
  | {
      id?: string;
      kind: "assistant_turn";
      role: "assistant";
      markdown: string;
      activity: SafeActivityEntry[];
      status: "streaming" | "complete" | "cancelled" | "failed";
    }
  | { id?: string; kind: "status"; role: "status"; tone: "neutral" | "error" | "cancelled"; text: string };
```

Keep `role` temporarily for CSS and compatibility during the refactor, but new behavior should branch on `kind`.

`AgentEvent` supports a safe activity event. For issue 008, keep activity display text constrained to allowlisted labels only. Do not add free-form provider summaries yet; issue 013 should decide whether Codex exposes user-facing summaries with safe provenance.

```ts
export type SafeAgentActivity =
  | { kind: "tool"; phase: "started"; label: "Tool started" }
  | { kind: "tool"; phase: "finished"; label: "Tool finished" }
  | { kind: "progress"; label: "Working" | "Reading" | "Searching" }
  | { kind: "error"; label: "Activity error" };

export type AgentEvent =
  | { type: "assistant.text.delta"; text: string }
  | { type: "assistant.activity"; activity: SafeAgentActivity }
  | { type: "assistant.done" }
  | { type: "assistant.cancelled" };
```

Session errors use typed protocol codes so bridge and extension behavior cannot drift:

```ts
export type SessionErrorCode =
  | "session_not_started"
  | "turn_in_flight"
  | "no_in_flight_turn"
  | "provider_error"
  | "provider_start_failed"
  | "provider_unavailable"
  | "unsafe_provider_event"
  | "unknown_error";
```

`session.error.code` stays optional for compatibility in this issue. Missing codes are treated as `unknown_error` by the extension. Unknown codes are rejected by the protocol parser; bridge code maps unknown internal failures to `unknown_error` before emitting. The extension treats `unknown_error` as terminal for the current turn if one is streaming, but it does not clear provider-started state.

Protocol validation rejects unknown activity kinds, private reasoning payload fields, extra fields, missing allowlisted labels, and malformed text. No issue-008 activity shape has a free-form field where a provider could place raw reasoning, stdout, prompts, or page content. Provider-originated events and bridge-to-extension envelopes must not accept raw `thought`, `chainOfThought`, `reasoning`, `prompt`, `pageContent`, `stdout`, or `stderr` fields. Valid extension-owned `session.send.prompt` and `session.send.pageContext` remain allowed only on `session.send`.

The side panel renders:

- user prompts as escaped compact right-aligned bubbles
- assistant turns with sanitized Markdown
- links with `target="_blank"` and `rel="noreferrer noopener"` only for allowed URL schemes
- Markdown images disabled so assistant output cannot auto-load remote resources from the extension UI
- fenced code blocks with a copy button
- safe activity inside a collapsed `Activity` disclosure
- operational errors as inline status cards with error styling
- `assistant.cancelled` marks the current assistant turn cancelled and appends a visible cancelled status card, while preserving partial assistant output

## What We're NOT Doing

- Not integrating Codex provider event taxonomy. Issue 013 maps Codex events into these safe protocol events.
- Not adding permission request UI. Issue 011 owns permission cards.
- Not implementing cancel controls. Issue 009 owns send-button cancel behavior and Escape.
- Not changing New Chat/reset lifecycle or reset-time provider replacement behavior. Issue 010 owns reset semantics.
- Not accepting provider reasoning summaries until issue 013 validates Codex event provenance.
- Not adding persistent transcript storage.
- Not rendering raw provider logs, raw stdout/stderr, raw chain-of-thought, or arbitrary HTML.
- Not showing full captured page context or DOM inside transcript entries.
- Not changing URL session key behavior.

## Implementation Approach

Grow the protocol first, then transcript reducers, then UI rendering. Keep renderer code behind small named components so `SidePanelView` remains a view instead of a Markdown-processing module.

Use proven Markdown/sanitization packages instead of hand-rolled parsing:

- `react-markdown`
- `remark-gfm`
- `rehype-sanitize`

Configure Markdown rendering to skip or sanitize raw HTML. Prefer not rendering raw HTML at all. Add a small `AssistantMarkdown` component that owns Markdown rendering, link props, code-block rendering, and copy behavior. Links should allow only `http:`, `https:`, and `mailto:`. Unsafe or malformed URLs should render without a clickable `href`. Disable Markdown images entirely so assistant content cannot trigger automatic network fetches.

Provider adapters own raw SDK event mapping. `BridgeSessionManager` must not understand raw provider taxonomy or decide whether a raw SDK string is safe to show. Adapters emit only safe bridge-facing turn events. The session manager validates those safe adapter outputs as defense in depth before sending them through the protocol. Invalid adapter outputs abort the current turn with a generic `session.error` such as `Provider emitted an unsafe event.` and must not echo the unsafe payload.

## Testing Strategy

Unit tests cover protocol validation and transcript reducer behavior.

Component tests cover Markdown sanitization, escaped user text, link attributes, code copy, collapsed activity, and status-card styling semantics.

Integration-style extension tests cover coordinator and URL session isolation through public module interfaces. These tests should emit bridge messages with matching and non-matching `clientSessionId` values.

## Performance Considerations

Assistant Markdown should render from the current assistant turn text as deltas arrive. Avoid reparsing unrelated transcript entries by keeping transcript entries immutable and rendering each entry as a keyed component.

Do not copy large code blocks unless the user clicks the copy button.

## Migration Notes

The transcript shape changes in two steps. Phase 2 keeps a temporary compatibility `text` field so reducer work can type-check before React rendering changes. Phase 4 moves rendering to `TranscriptView`; after that, new view code should branch on `kind` and stop depending on assistant-turn compatibility text.

Adding Markdown packages changes `pnpm-lock.yaml`. That lockfile update belongs with Phase 4 implementation.
