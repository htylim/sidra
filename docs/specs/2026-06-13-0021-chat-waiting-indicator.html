Spec 021: Chat Waiting Indicator

Add an animated, accessible waiting indicator to Sidra's side panel whenever the active URL session is waiting for the bridge or provider turn to finish. The indicator must use the same lifecycle source as the existing Send and Cancel controls.

Overview

When a user sends a prompt, Sidra currently disables prompt controls and eventually swaps the Send button to Cancel. The transcript may show only Session started until the first provider event arrives. This creates a silent wait state during startup, follow-up prompts, provider work, and cancel acknowledgement.

The implementation should render a small assistant-side waiting row with short copy and animated dots. It should appear while the active session is blocked from sending because a prompt is queued or a turn is running and Sidra is waiting on the bridge or provider. It should disappear when the lifecycle returns to the same idle state that enables writing and sending again. It should not appear when a permission card is waiting for user action.

UX Research

Source Useful guidance Decision for Sidra
NN/g response time limits Feedback matters after about one second. For unknown duration, a spinner, dots, or similar running feedback is acceptable. Use an indeterminate indicator. Do not fake percent progress because the bridge cannot know completion percent.
Carbon loading component Loading indicators show that a request is being processed. Small inline indicators work for localized loading near the triggering action. Use one small inline indicator in the chat transcript, not a full-panel overlay.
WAI-ARIA status role role="status" is advisory, polite live-region feedback and should not take focus. Expose the waiting row as a polite status. Do not move focus.
WCAG pause, stop, hide guidance Moving or blinking content can distract users and screen readers. Avoid persistent distracting motion. Animate only small dots, avoid flashing, and provide a prefers-reduced-motion fallback.

Recommended pattern: an assistant-side inline status row with text plus an animated dot trio. Copy: Waiting for response during queued and running states, and Cancelling after cancel is requested.

Desired State

  • After sending a prompt, a visible waiting indicator appears in the transcript area before the first bridge or provider response is visible.
  • The same indicator appears for follow-up prompts and quick actions.
  • The indicator remains visible while Sidra is waiting for the active bridge turn to finish.
  • After Cancel is clicked, the indicator changes to Cancelling until the terminal bridge event clears the running turn.
  • When a permission request is pending, the permission card is the visible blocking state and the waiting indicator is hidden.
  • The indicator disappears when the active session can accept typing and sending again.
  • The indicator is not stored as a transcript entry and does not appear in completed transcript history.
  • The indicator is accessible through a polite status region and does not steal focus.
  • Animation respects prefers-reduced-motion: reduce.

What We Are Not Doing

  • No bridge protocol changes.
  • No Native Messaging bridge changes.
  • No provider-specific progress events.
  • No fake progress bar or percent complete.
  • No persistent transcript entry for Waiting for response or Cancelling.
  • No full-panel overlay that blocks reading prior transcript content.
  • No new user preference for loading style.

Implementation Approach

  1. Keep lifecycle ownership in BridgeSessionCoordinator, UrlSessionStore, and SidePanelController.
  2. Add a small view-level work-state helper in apps/extension/src/side-panel-view.tsx.
  3. Use that helper for prompt disabling, Send or Cancel rendering, and waiting indicator state so those affordances cannot drift.
  4. Pass the derived waiting state to TranscriptView.
  5. Render one ephemeral waiting row after transcript entries while the derived state is active.
  6. Replace the static Streaming text with the new waiting indicator so there is only one visible progress affordance.
  7. Style the indicator as an assistant-side inline row with a fixed-width dot group and reduced-motion fallback.
Recommended TypeScript shape 
type ChatWorkState =
  | { kind: "idle" }
  | { kind: "queued_startup" }
  | { kind: "running" }
  | { kind: "cancelling" };

type ChatWaitingState =
  | { kind: "idle" }
  | { kind: "waiting_for_response"; label: "Waiting for response" }
  | { kind: "cancelling"; label: "Cancelling" };

function deriveChatWorkState(activeSession: SidePanelSnapshot["activeSession"]): ChatWorkState {
  if (activeSession.pendingPromptCount > 0) return { kind: "queued_startup" };
  if (activeSession.turnInFlight && !activeSession.canCancelTurn) {
    return { kind: "cancelling" };
  }

  if (activeSession.turnInFlight) return { kind: "running" };
  return { kind: "idle" };
}

function deriveChatWaitingState(activeSession: SidePanelSnapshot["activeSession"]): ChatWaitingState {
  if (hasPendingPermissionRequest(activeSession.transcript)) return { kind: "idle" };

  const workState = deriveChatWorkState(activeSession);
  if (workState.kind === "queued_startup" || workState.kind === "running") {
    return { kind: "waiting_for_response", label: "Waiting for response" };
  }
  if (workState.kind === "cancelling") return { kind: "cancelling", label: "Cancelling" };
  return { kind: "idle" };
}

function hasPendingPermissionRequest(transcript: SidePanelSnapshot["activeSession"]["transcript"]): boolean {
  return transcript.some((entry) => entry.kind === "permission_request" && entry.status === "pending");
}

The work-state helper intentionally uses the same fields that already disable prompt entry and select the Send or Cancel button. Prompt disabling should be computed from chatUnavailable || workState.kind !== "idle", and Cancel should be enabled only for workState.kind === "running". It must not inspect bridge messages directly.

Risks

Risk Mitigation
Indicator lingers after final response. Do not store it in transcript. Derive it from pendingPromptCount and turnInFlight.
Duplicate progress affordances create noise. Replace the static Streaming text. Keep exactly one waiting indicator.
Cancel-requested state looks idle because Cancel is disabled. Show Cancelling while turnInFlight is true and canCancelTurn is false.
Animation distracts or violates user motion preferences. Use small dot motion only and disable animation with prefers-reduced-motion: reduce.
View learns too much about bridge protocol sequencing. Use only active-session snapshot fields. Do not inspect protocol messages or bridge event types in React.
Waiting UI appears while the app needs user permission input. Suppress the waiting indicator when the transcript contains a pending permission_request.
