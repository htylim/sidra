# Provider Session Reset And Close Lifecycle Implementation Plan

## Overview

Fix the local-only reset scaffold so provider-affecting lifecycle operations are represented in the protocol and handled by the bridge. New Chat must not leak provider sessions or leave in-flight turns running.

## Current State Analysis

The extension now routes side-panel behavior through `SidePanelController`, which composes `BridgeConnection` and `BridgeSessionCoordinator`. `session-client.ts` has been removed. `BridgeSessionCoordinator` owns the active extension-side session snapshot and prompt queue, and it still has only local reset-shaped behavior; New Chat must not use that as the provider lifecycle implementation.

The protocol currently has no `session.reset` or `session.close` commands. The bridge keeps provider sessions inside `createBridge`, but has no bridge-owned reset/close command handling and no retained in-flight controller that reset/close can abort.

This is a P1 architecture issue because New Chat is a V1 product behavior. If implemented locally, it would make the UI look reset while the bridge provider session still exists.

## Desired End State

- Protocol supports `session.reset` and `session.close`.
- Bridge reset closes the old provider session, aborts any in-flight turn, creates a fresh provider session for the same `clientSessionId`, and emits a new `session.started`.
- Bridge close closes and removes the provider session for one `clientSessionId`.
- Extension `newChat` clears only the active session's local transcript/draft/context and sends the correct bridge lifecycle command when provider state may exist.
- No local-only provider lifecycle actions remain.

### Key Discoveries

- Extension composition now lives in `apps/extension/src/side-panel-controller.ts`.
- Extension session state and prompt queueing live in `apps/extension/src/bridge/session-coordinator.ts`.
- `apps/extension/src/bridge/session-client.ts` has been deleted and must not be referenced as the current reset path.
- Current protocol includes only `session.start`, `session.send`, and `heartbeat` in `packages/protocol/src/index.ts:13`.
- Current bridge provider sessions are stored in `apps/bridge/src/index.ts`, with no reset/close dispatch path or retained per-session in-flight abort controller.
- The PRD defines `session.reset` and `session.close` in the Bridge Protocol section.
- ADR-003 requires provider-affecting lifecycle actions to be represented in protocol and handled by the bridge before UI exposes them.

## What We're NOT Doing

- Not implementing URL session switching.
- Not implementing session-scoped provider permissions.
- Not implementing heartbeat cleanup.
- Not replacing the mock provider.
- Not changing the visual design of New Chat beyond wiring the existing button.

## Implementation Approach

Build on the current `SidePanelController` and `BridgeSessionCoordinator` boundary. It also benefits from `BridgeSessionManager` in `spec-001-02.md`; if that manager does not exist yet, create the reset/close behavior in the same manager shape described there instead of leaving provider lifecycle in the raw bridge message switch. Implement protocol support first, bridge lifecycle second, extension command third, and UI wiring last.

## Phase 2: Implement Bridge Reset And Close Semantics

### Overview

Make the bridge own provider-session cleanup and fresh session creation.

### Why This Phase Can Be Validated Independently

The bridge manager can be tested with fake provider sessions that record `close()` calls and abort signals.

### Changes Required

#### 1. Tests (RED)

**File**: `apps/bridge/src/session-manager.test.ts`

Add these tests:

```ts
describe("BridgeSessionManager reset and close", () => {
  it("reset closes the old provider session and creates a fresh provider session");
  it("reset emits session.started for the same clientSessionId");
  it("reset aborts an in-flight turn before creating the fresh provider session");
  it("close aborts an in-flight turn and closes the provider session");
  it("close removes the provider session so later send returns session_not_started");
  it("reset and close do not affect another client session");
  it("close is idempotent when the client session does not exist");
});
```

#### 2. Implementation (GREEN)

**File**: `apps/bridge/src/session-manager.ts`

Add:

- `resetSession(clientSessionId)`
- `closeSession(clientSessionId)`

Reset behavior:

1. If a session exists, abort any in-flight turn and await close.
2. Create a fresh provider session for the same `clientSessionId` and prior provider ID. In V1, the only provider is `codex`.
3. Emit `session.started` with a new `bridgeSessionId`.

Close behavior:

1. If a session exists, abort any in-flight turn.
2. Await provider `close()`.
3. Remove the managed session.
4. Do not emit prompt or page content in errors.

**File**: `apps/bridge/src/index.ts`

Dispatch `session.reset` and `session.close` to the manager.

- Clear active local transcript, draft, context state, pending prompt queue, and running state.

Keep coordinator reset behavior as extension-local state cleanup only. Add a protocol-backed command path for provider reset, or expose the minimal coordinator method the controller needs to post `session.reset`. ## Testing Strategy

Protocol tests prove command shape. Bridge manager tests prove provider cleanup and in-flight abort behavior. Controller tests prove extension New Chat does not become local-only. UI/boundary tests prove the visible button reaches the command.

## Performance Considerations

Reset and close should operate only on one provider session. They should not scan or block unrelated sessions except for constant-time map lookup.

## Migration Notes

Implement this after or alongside the bridge session manager. Do not recreate `BridgeSessionClient.reset()` as the API. New Chat must call the controller and result in a bridge-owned `session.reset` when provider state may exist.
