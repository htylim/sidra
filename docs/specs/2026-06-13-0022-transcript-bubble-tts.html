Implementation Plan

Transcript Bubble Text To Speech

Add a hover and keyboard-focus playback button to Sidra transcript bubbles so the user can listen to a user prompt or completed assistant response without reading it.

Overview Current State Research Desired State Approach Phase 1 Phase 2 Phase 3 Phase 4 Testing References

Overview

V1 uses Chrome's native chrome.tts extension API. This avoids API keys, separate billing, network latency, provider secrets, and bridge/protocol changes. Hosted TTS providers stay out of scope for V1.

The feature belongs in apps/extension. Transcript text already exists in the extension snapshot, and native speech is browser-side UI behavior over that text.

  • SidePanelView receives a derived snapshot and passes transcript entries into TranscriptView: apps/extension/src/side-panel-view.tsx:110.
  • TranscriptView renders user messages and assistant turns: apps/extension/src/transcript-view.tsx:21, apps/extension/src/transcript-view.tsx:81, apps/extension/src/transcript-view.tsx:117.

Research

Option Fit Cost and access Decision
chrome.tts Best fit for V1. Extension API. Supports speak, stop, queueing, language, rate, events. No separate account, no OpenAI API key, no network billing. Use for V1.
window.speechSynthesis Works in web pages, but less extension-specific than chrome.tts. No separate account. Do not use while chrome.tts is available.
OpenAI Speech API Good hosted TTS quality. Uses POST /v1/audio/speech and gpt-4o-mini-tts. OpenAI APIs are billed separately from ChatGPT plans. Needs API key and billing. Defer to later provider work.
Realtime voice Built for live voice agents. Too much state and transport for reading one bubble. Separate API billing and realtime session complexity. Out of scope.
ElevenLabs, Amazon Polly, Google Cloud TTS Common hosted TTS options, useful if native quality is not enough. Each needs provider account, billing, secrets, and network handling. Out of scope for V1.
Provider notes
  • OpenAI's text-to-speech guide recommends gpt-4o-mini-tts as the newest reliable TTS model.
  • OpenAI's audio guide separates bounded request-based TTS from realtime voice sessions.
  • OpenAI API pricing FAQ states APIs are billed separately from ChatGPT Plus, Business, Enterprise, and Edu. Pro is a ChatGPT plan and should be treated the same product boundary for Sidra API usage.
  • Provider TTS must not put an API key in the extension UI. It would require a bridge-owned or local-secret-backed provider boundary.

Desired End State

  1. Hovering a speakable transcript bubble reveals an icon button.
  2. Keyboard focus can reach the same button, and focus makes the button visible.
  3. Clicking the button speaks that bubble's text through native Chrome TTS.
  4. Clicking a different bubble stops current speech and starts the selected bubble.
  5. Clicking the active bubble's button stops speech.
  6. User message bubbles are speakable.
  7. Completed, failed, or cancelled assistant response bubbles with visible text are speakable.
  8. Streaming assistant responses are not speakable until the response stops changing.
  9. Status cards, permission cards, activity details, and empty assistant turns are not speakable in V1.
  10. Quick-action user messages speak the full prompt text, not just the collapsed label.
  11. Speech stops on New Chat, active page change, side panel unload, and controller shutdown.
  12. No OpenAI API, no hosted TTS provider, and no bridge/protocol changes are added in V1.

What We're NOT Doing

  • No OpenAI TTS integration.
  • No ElevenLabs, Amazon Polly, Google Cloud, or other hosted provider.
  • No API key UI, secret storage, or local credential flow.
  • No bridge protocol changes.
  • No TTS settings page for voice, rate, pitch, or provider selection.
  • No automatic reading of every assistant response.
  • No live reading of streaming deltas.
  • No transcript persistence changes.

Implementation Approach

Add a small browser-side speech boundary under apps/extension/src/transcript-speech.ts. It wraps chrome.tts, exposes an application-level interface, owns speech state, and hides event ordering details from React.

Integrate that boundary into SidePanelController. The React view receives speech state and intent callbacks, but it does not call chrome.tts or read browser APIs directly.

Render a stable icon button inside speakable transcript bubbles. CSS reveals the button on hover and focus-within. The button is always present in the DOM for keyboard and screen reader access.

Testing Strategy

Layer What to test Key edge cases
Unit TranscriptSpeechController gateway calls, state, final events, errors, shutdown. Blank text, stale events, interrupted playback, missing voice errors.
Controller Snapshot speech state and application commands. New Chat, page switch, shutdown.
React Button presence, payload selection, active stop state, excluded entries. Quick-action labels, assistant markdown vs plain text, streaming responses.
CSS Hover/focus reveal and layout stability selectors. Disabled or active state, keyboard focus.
Manual browser Native audio output and extension permission behavior. No TTS voices, OS audio muted, browser-specific voice behavior.

Performance Considerations

  • Native TTS avoids network requests and audio file caching.
  • Do not precompute speech or enumerate voices on every render.
  • Transcript buttons should be ordinary DOM controls with CSS opacity, not per-message timers.
  • Controller state should emit only when speech state changes.

Migration Notes

Existing transcript state does not need migration. The manifest permission change affects extension install or reload prompts in Chromium. Existing local Native Messaging setup is unchanged.
