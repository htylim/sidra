# Extension Session Boundary Refactor Implementation Plan

## Overview

Refactor the issue-001 extension scaffold so session orchestration is behind a deep non-React application boundary. The goal is to preserve the current mock chat behavior while preventing `BridgeSessionClient` from becoming the owner of transport, transcript, provider startup, UI state, URL sessions, capture, cancellation, and errors.

## Current State Analysis

`apps/extension/src/bridge/session-client.ts` currently owns the Chrome Native Messaging port, connection state, provider session startup, queued prompts, transcript messages, error status messages, retry, and a React-facing snapshot shape. `apps/extension/src/side-panel.tsx` creates the singleton directly and renders that snapshot with `useSyncExternalStore`.

This is acceptable for a first scaffold, but it is too shallow for the next V1 work. URL-scoped sessions need page-keyed state, capture needs active-tab orchestration, New Chat needs provider lifecycle commands, and cancellation needs per-session running-turn state. If those behaviors are added to the current class, the extension boundary will become hard to test and hard to change.

## Desired End State

The extension has a deep application boundary:

- React renders a `SidePanelSnapshot` and calls controller commands.
- `SidePanelController` is the public extension application API used by the view.
- `BridgeConnection` owns Native Messaging connection and raw protocol IO.
- A session coordinator owns provider startup sequencing and prompt queueing for one client session.
- Transcript state is represented by explicit extension-domain types, not by a generic `ChatMessage` owned by the transport helper.
- Existing issue-001 behavior still passes: prompt starts a session, waits for `session.started`, sends the prompt, and renders mock assistant text.

### Key Discoveries

- `BridgeSessionClient` exposes UI state directly through `BridgeSessionState` in `apps/extension/src/bridge/session-client.ts:7`.
- `BridgeSessionClient.sendPrompt` appends user transcript entries before it posts bridge messages in `apps/extension/src/bridge/session-client.ts:62`.
- `BridgeSessionClient.ensurePort` owns the Chrome Native Messaging port in `apps/extension/src/bridge/session-client.ts:121`.
- `SidePanel` imports the bridge client directly in `apps/extension/src/side-panel.tsx:3`.
- ADR-001 says side panel UI should call application-level commands, and ADR-003 now defines the expected module ownership.

## What We're NOT Doing

- Not implementing URL-scoped sessions.
- Not implementing page capture or extraction.
- Not implementing New Chat protocol reset.
- Not implementing cancellation.
- Not changing the visible side panel design.
- Not replacing the mock provider.

## Implementation Approach

Move behavior behind a stable controller interface one vertical slice at a time. Tests must be written against public extension boundaries first, then implementation should move logic out of `BridgeSessionClient` until that file is deleted or reduced to a compatibility wrapper. The refactor is successful only if behavior remains the same and new modules have narrower ownership.

## Phase 1: Define The Side Panel Controller Contract

### Overview

Introduce the public non-React application boundary that the side panel will use.

### Why This Phase Can Be Validated Independently

The controller can wrap the existing `BridgeSessionClient` at first. That allows tests to prove the public command/snapshot contract before internals are moved.

### Changes Required

> TDD ordering: list tests first, implementation second.

#### 1. Tests (RED)

**File**: `apps/extension/src/side-panel-controller.test.ts`

Add these tests:

```ts
describe("SidePanelController", () => {
  it("queues the first prompt until the bridge session starts");
  it("flushes queued prompts in order after session.started");
  it("renders user and assistant transcript entries through the controller snapshot");
  it("rejects blank prompts without changing the snapshot");
  it("surfaces bridge errors as setup state without exposing protocol details to the view");
});
```

Each test should use a fake Native Messaging port and interact only with controller commands and snapshots.

#### 2. Implementation (GREEN)

**File**: `apps/extension/src/side-panel-controller.ts`

Create:

```ts
export type SidePanelSnapshot = {
  bridge: {
    connected: boolean;
    ready: boolean;
    setupError?: string;
  };
  activeSession: {
    clientSessionId: string;
    transcript: TranscriptEntry[];
    pendingPromptCount: number;
    sessionStarted: boolean;
    starting: boolean;
  };
};

export type SidePanelController = {
  getSnapshot(): SidePanelSnapshot;
  subscribe(listener: () => void): () => void;
  sendPrompt(prompt: string): boolean;
  retryBridge(): void;
};
```

Initial implementation may delegate to `BridgeSessionClient`, but the exported interface must be the controller contract, not the old bridge client state.

**File**: `apps/extension/src/bridge/session-client.ts`

Keep behavior unchanged. Add no new responsibilities.

## Phase 2: Extract Native Messaging Transport

### Overview

Move Chrome Native Messaging connection ownership out of session orchestration.

### Why This Phase Can Be Validated Independently

### Changes Required

#### 1. Tests (RED)

Add these tests:

```ts
describe("BridgeConnection", () => {
  it("connects lazily on the first posted message");
  it("reuses the current port for later posts");
  it("publishes bridge.ready as connection readiness");
  it("publishes disconnect state without mutating transcript state");
  it("returns a failed post result when connectNative throws");
  it("returns a failed post result when port.postMessage throws");
});
```

#### 2. Implementation (GREEN)

**File**: `apps/extension/src/bridge/connection.ts`

Create `BridgeConnection` with:

- `getSnapshot()`
- `subscribe(listener)`
- `post(message: ExtensionToBridge): PostResult`
- `retry()`
- `disconnect()` for cleanup tests and later side-panel shutdown.

Move `NativeBridgePort` and `SIDRA_NATIVE_HOST` here.

**File**: `apps/extension/src/bridge/session-client.ts`

Replace direct `connectNative` ownership with the new `BridgeConnection` or leave a compatibility wrapper that delegates to it.

## Phase 3: Extract Session Coordination And Transcript State

### Overview

Move provider-session startup sequencing, pending prompt queueing, and transcript updates into extension-domain modules.

### Why This Phase Can Be Validated Independently

The session coordinator can be tested with a fake `BridgeConnection` and protocol messages without rendering React.

### Changes Required

#### 1. Tests (RED)

**File**: `apps/extension/src/bridge/session-coordinator.test.ts`

Add these tests:

```ts
describe("BridgeSessionCoordinator", () => {
  it("posts session.start before the first session.send");
  it("queues prompts while waiting for session.started");
  it("flushes queued prompts in original order");
  it("reuses the started provider session for later prompts");
  it("ignores session.started for another clientSessionId");
  it("appends assistant deltas to the transcript without knowing React");
  it("clears pending startup state after bridge.error");
});
```

**File**: `apps/extension/src/transcript.test.ts`

Add these tests:

```ts
describe("transcript reducer", () => {
  it("adds escaped user prompt entries");
  it("adds assistant text delta entries");
  it("adds operational status entries");
});
```

#### 2. Implementation (GREEN)

**File**: `apps/extension/src/bridge/session-coordinator.ts`

Create a one-client-session coordinator that owns:

- `clientSessionId`
- `providerId`
- `sessionStarted`
- `starting`
- `pendingPrompts`
- mapping bridge events to session state changes.

It should depend on `BridgeConnection` through a narrow interface such as:

```ts
type ProtocolTransport = {
  post(message: ExtensionToBridge): PostResult;
  subscribeToMessages(listener: (message: BridgeToExtension) => void): () => void;
};
```

**File**: `apps/extension/src/transcript.ts`

Create explicit transcript entry types and reducer/helper functions.

**File**: `apps/extension/src/bridge/session-client.ts`

Delete it if all callers have moved, or reduce it to a temporary compatibility wrapper with no direct Chrome port ownership and no direct transcript mutation.

## Phase 4: Rewire Side Panel To The Controller Boundary

### Overview

Make the React side panel consume the controller contract instead of bridge internals.

### Why This Phase Can Be Validated Independently

The UI should keep the same visible behavior while import-boundary tests prove React does not reach into transport/session internals.

### Changes Required

#### 1. Tests (RED)

**File**: `apps/extension/src/side-panel-boundary.test.ts`

Add these tests:

```ts
describe("side panel architecture boundary", () => {
  it("keeps side-panel.tsx free of bridge/session-client imports");
  it("keeps side-panel view files free of chrome runtime usage");
  it("creates Chrome-backed dependencies only in the side panel composition root");

```

The test may read source files to enforce the architecture boundary. This is an explicit architecture test, not a substitute for behavior tests.

#### 2. Implementation (GREEN)

**File**: `apps/extension/src/side-panel.tsx`

Use `createChromeSidePanelController()` from the controller module and pass its snapshot/commands into a pure view component.

**File**: `apps/extension/src/side-panel-view.tsx`

Extract a pure view component:

```ts
export function SidePanelView(props: {
  snapshot: SidePanelSnapshot;
  onSendPrompt(prompt: string): boolean;
  onRetryBridge(): void;

```

## Testing Strategy

Unit tests cover controller behavior, transport behavior, session coordination, transcript state, and architecture import boundaries. Existing smoke/e2e coverage continues to prove the side panel renders.

## Performance Considerations

No meaningful runtime cost is expected. The refactor should reduce unnecessary React coupling and make future URL session updates cheaper by narrowing snapshot changes.

## Migration Notes

This is an internal refactor. Preserve the current public UI. Delete compatibility wrappers as soon as callers are moved.
