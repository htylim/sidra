# Current URL Session New Chat Reset Implementation Plan

## Overview

Implement issue 010: `New Chat` resets only the currently visible URL session, closes or replaces that session's provider thread, clears local visible session state, and returns the active page to the empty state without disturbing other URL sessions.

Most of the low-level reset path already exists. This plan hardens the extension-owned URL-session behavior, fills missing acceptance coverage, and keeps future permission approval clearing attached to the same session reset boundary.

## Current State Analysis

Issue 010 requires `New Chat` to send the correct reset or close/start sequence for the active `clientSessionId`, clear the current transcript, draft, context state, running state, and session-scoped approvals, create a fresh provider session for the same page key, preserve other URL sessions, and cover reset during and after running turns: `docs/issues/issue-010.md:13`.

Current code already contains important support:

- `session.reset` and `session.close` are protocol commands with `clientSessionId` validation: `packages/protocol/src/index.ts:62`.
- `BridgeSessionManager.resetSession()` replaces the provider session for one `clientSessionId`; `closeManagedSession()` aborts an in-flight turn before closing the provider session: `apps/bridge/src/session-manager.ts:136`, `apps/bridge/src/session-manager.ts:213`.
- `BridgeSessionManager` serializes lifecycle operations per client session while allowing unrelated sessions to progress independently: `apps/bridge/src/session-manager.ts:155`.
- `BridgeSessionCoordinator.newChat()` clears pending submissions and running/cancel state, posts `session.reset` when provider state may exist, and suppresses stale startup or turn results from the old provider thread: `apps/extension/src/bridge/session-coordinator.ts:209`.
- `UrlSessionStore.newChat()` routes to only the active URL session and clears draft, context state, and capture mode before delegating to the coordinator: `apps/extension/src/url-session-store.ts:198`.
- `SidePanelController.newChat()` currently forwards directly to `UrlSessionStore.newChat()`: `apps/extension/src/side-panel-controller.ts:265`.
- The PRD says `New Chat` must reset only the visible URL session, close the current provider thread, clear transcript, draft, and session-scoped permissions, start a new provider thread for the same page key, and show empty-state quick actions again: `docs/prd-v1.md:221`.

Important gaps:

- Tests cover pieces of reset behavior, but acceptance coverage is scattered and misses some end-to-end assertions: same page key after reset, other-session draft/transcript/context/running preservation from the controller surface, empty-state quick actions after reset, reset while cancellation is pending, and reset after a completed running turn.
- There is no permission request UI or provider permission protocol yet, but issue 010 still requires a reset-owned session approval clearing boundary. The missing piece is not permission cards; it is a small extension-owned session-scoped approval state/hook that New Chat can clear and issue 011 can later use.
- Current view tests mostly prove the `New Chat` button is wired, not that the active page returns to the empty-state quick-action experience after reset.

## Desired End State

`New Chat` behavior is owned by the existing boundaries:

- `SidePanelView` only invokes `onNewChat`.
- `SidePanelController` exposes `newChat()` and recomputes the view snapshot after reset.
- `UrlSessionStore` chooses the active URL session and clears extension-owned session state for that record only.
- `BridgeSessionCoordinator` owns provider reset sequencing and stale event suppression for that `clientSessionId`.
- `BridgeSessionManager` owns provider-session replacement, abort, and close.

For the active URL session, `newChat()` must:

- Clear transcript, draft prompt, context state, pending queued prompt count, running state, cancel state, and reset capture mode to readable.
- Post `session.reset` for the active `clientSessionId` when provider state may exist.
- Not post reset for a never-started empty session.
- Keep the same page key and `clientSessionId` after reset.
- Wait for the fresh `session.started` event before sending a later queued prompt.
- Ignore stale startup, terminal, stream, and cancel reconciliation events from the old provider turn.
- Show the empty state and visible quick actions again when settings and bridge/page state allow them.

For other URL sessions, `newChat()` must preserve transcript, draft prompt, context state, pending/running state, provider session, and cancel state.

For permissions, issue 010 must add a minimal production-facing extension-owned approval clearing boundary without adding permission UI or bridge protocol. A URL session needs a session-scoped approval owner with a real contract, not test-only state:

```ts
export type SessionApproval = {
  permissionKey: string;
  decision: "allow_for_session";
};

export type SessionApprovalState = {
  grant(approval: SessionApproval): void;
  has(permissionKey: string): boolean;
  list(): SessionApproval[];
  clear(): void;
};
```

This owner belongs to the URL-session lifecycle and is intentionally not exposed to React in issue 010. It must still be production-facing through the non-React application boundary:

```ts
type UrlSessionStore = {
  grantActiveSessionApproval(approval: SessionApproval): void;
  hasActiveSessionApproval(permissionKey: string): boolean;
  listActiveSessionApprovals(): SessionApproval[];
};
```

The controller does not need to expose these methods to React yet, but issue 011 must be able to use them from the extension application layer when provider permission responses are added. `newChat()` must clear approvals for the active URL session only.

Stale same-session provider events after reset need a concrete ordering contract because protocol v2 events do not include a provider-session generation. Issue 010 should not change protocol. Instead, the bridge must guarantee that `resetSession(clientSessionId)` waits for old in-flight provider work to finish or abort, closes the old provider session, and only then emits the fresh `session.started`. The extension may ignore old events that arrive before the fresh `session.started`; after the fresh start and a new prompt, old same-`clientSessionId` events are not distinguishable and must be prevented by the bridge/session-manager ordering guarantee.

## What We're NOT Doing

- No new protocol version. `session.reset` and `session.close` already exist.
- No React-owned reset sequencing.
- No provider SDK-specific reset logic. Provider adapters only implement `close()`.
- No permission request UI, permission protocol, provider permission handling, or approval policy. Issue 011 owns those.
- No persistent transcript or draft storage.
- No global reset of all URL sessions.
- No new `clientSessionId` for the same page key during New Chat unless a later architecture decision explicitly changes that identity model.

## Implementation Approach

Use the current deep-module boundaries. Start by adding missing tests around `BridgeSessionCoordinator` and `UrlSessionStore`, then add controller and view tests that prove the product behavior from the side panel surface. Keep code changes small because most reset behavior already exists.

Do not write duplicate tests only to satisfy a red step. When a listed behavior already has a passing test, keep it as regression coverage and add only the missing failing assertion named in this spec. The RED requirement applies to newly added tests only.

If tests reveal a gap, fix the owning boundary:

- stale protocol sequencing belongs in `BridgeSessionCoordinator`
- active URL-session selection and local state clearing belong in `UrlSessionStore`
- session-scoped approval clearing belongs in an extension session lifecycle boundary, owned by `UrlSessionStore` or a small production collaborator on each URL-session record
- durable ownership docs must be updated if a new approval owner is added
- snapshot refresh and command routing belong in `SidePanelController`
- provider abort/replacement belongs in `BridgeSessionManager`
- rendering the empty state belongs in `SidePanelView`

## Testing Strategy

Unit and integration tests should cover reset at the owning boundaries:

- Protocol tests only need to keep validating existing `session.reset` and `session.close` shapes.
- Bridge tests prove provider abort/replacement and cross-session isolation.
- Coordinator tests prove reset protocol sequencing, stale event suppression, and local transcript/running cleanup.
- URL-session store tests prove active-session-only clearing and inactive-session preservation.
- Controller/view tests prove user-visible empty-state behavior.

## Performance Considerations

Reset should stay O(1) for the active URL session in extension code. Do not scan all sessions for New Chat. Bridge operation queues remain per `clientSessionId`, so one session reset must not block unrelated provider turns.

## Migration Notes

No persisted data migration is required. Sessions are in memory. Existing `clientSessionId` values remain valid across New Chat for the same page key.
