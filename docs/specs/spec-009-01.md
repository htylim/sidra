# Cancel In-Flight Responses Implementation Plan

## Overview

Implement issue 009: while one URL session has a running provider turn, Sidra shows a Cancel control for that session, blocks duplicate sends in that same session, allows other URL sessions to continue running, and routes cancellation through `session.cancel` so the bridge aborts provider work and the transcript preserves any partial assistant output.

This is mostly extension-owned application behavior. The protocol and bridge already contain the cancellation command and provider abort lifecycle, so this plan adds the missing extension command, active URL-session routing, running-state snapshot, and view behavior.

## Current State Analysis

Issue 009 requires visible cancel behavior, same-session duplicate-send prevention, cross-session concurrency, Escape cancellation, preserved partial output, and a cancelled marker: `docs/issues/issue-009.md:13`.

Current code already has some lower-level support:

- `packages/protocol` defines and validates `session.cancel`: `packages/protocol/src/index.ts:62`, `packages/protocol/src/index.ts:179`.
- `BridgeSessionManager.cancelTurn()` aborts the target session's active `AbortController`, waits for provider send cleanup, and emits `assistant.cancelled`: `apps/bridge/src/session-manager.ts:109`.
- Bridge sessions track `inFlight` per `clientSessionId`, reject a second send for the same session with `turn_in_flight`, and do not serialize unrelated sessions behind one global turn: `apps/bridge/src/session-manager.ts:28`, `apps/bridge/src/session-manager.ts:71`, `apps/bridge/src/session-manager.ts:155`.
- `BridgeSessionCoordinator` already tracks a local `turnInFlight` after a `session.send` posts, rejects duplicate sends only after that point, marks `assistant.cancelled` as terminal, preserves partial assistant output, and appends the cancelled status card: `apps/extension/src/bridge/session-coordinator.ts:85`, `apps/extension/src/bridge/session-coordinator.ts:124`, `apps/extension/src/bridge/session-coordinator.ts:324`.
- `BridgeSessionCoordinator` still queues multiple prompts before `session.started`, so issue 009 must close that gap at the coordinator/store boundary instead of relying only on React controls: `apps/extension/src/bridge/session-coordinator.ts:160`.
- `UrlSessionStore` owns active URL-session routing and exposes per-session transcript, pending count, `sessionStarted`, and `starting`, but it has no cancel command and no explicit running flag: `apps/extension/src/url-session-store.ts:31`, `apps/extension/src/url-session-store.ts:147`.
- `SidePanelController` exposes application commands to React, but it has no `cancelTurn()` command and no `activeSession.turnInFlight` or `canCancelTurn` snapshot field: `apps/extension/src/side-panel-controller.ts:52`, `apps/extension/src/side-panel-controller.ts:278`.
- `SidePanelView` always renders a `Capture + Send` button and only handles Enter in the textarea; it has no Cancel button state and no Escape handler: `apps/extension/src/side-panel-view.tsx:28`, `apps/extension/src/side-panel-view.tsx:105`, `apps/extension/src/side-panel-view.tsx:140`.
- `captureAndSend()` captures page context before calling `UrlSessionStore.sendPromptWithContext()`, so a user can start an expensive capture while a turn is already running unless the controller/view gate it before capture begins: `apps/extension/src/side-panel-controller.ts:191`.

## Desired End State

`BridgeSessionCoordinatorSnapshot` exposes:

```ts
turnInFlight: boolean;
canCancelTurn: boolean;
```

`BridgeSessionCoordinator` must reject new sends whenever the same URL session has a pending startup prompt, has a provider turn in flight, or has a cancel request waiting for a terminal bridge event. It must not queue a second prompt behind the first startup prompt. A reset-owned `starting: true` state with no pending prompt is different: after New Chat posts `session.reset`, the user may queue one fresh prompt behind the reset `session.started`.

`canCancelTurn` is true only after a `session.send` has been posted to the bridge, before cancellation has been requested, and before a terminal `assistant.done`, `assistant.cancelled`, or terminal error clears the turn. It is false while a first prompt is only queued behind `session.start`; that queued-start state still blocks duplicate sends, but there is no provider turn to cancel yet.

`BridgeSessionCoordinator.cancelTurn()` posts:

```ts
{ type: "session.cancel", version: 2, clientSessionId }
```

It returns `false` and does not post if no provider turn is in flight, the bridge session has not started, cancellation has already been requested, or the transport post fails. Post failure should be visible as an inline error status tied to the current URL session. Successful posting should leave `turnInFlight` true until the bridge emits `assistant.cancelled`, `assistant.done`, or a terminal `session.error`; this avoids letting the UI send another prompt before the provider has actually stopped. It should also set an internal `cancelRequested` state that makes `canCancelTurn` false until that terminal event arrives. Repeated Cancel clicks or Escape presses must not post a second `session.cancel`. Because Native Messaging dispatch can race, the coordinator must also suppress a stale `no_in_flight_turn` error that arrives after a cancel request was already reconciled by `assistant.done` or `assistant.cancelled`.

`UrlSessionStore` routes `cancelActiveTurn()` to only the active URL session. It preserves other URL sessions and their running state.

`SidePanelController` exposes:

```ts
cancelTurn(): boolean;
```

The active session snapshot includes `turnInFlight` and `canCancelTurn`. The controller must also reject `sendPrompt()`, `captureAndSend()`, and `sendQuickAction()` before capture starts when the active session already has a queued prompt or running provider turn. This keeps duplicate sends from adding context markers, clearing drafts, or launching capture work while another turn owns the session.

`SidePanelView` renders the primary composer control as:

- `Capture + Send` when no provider turn is running.
- `Cancel` when `activeSession.turnInFlight` is true.

While a turn is running, the textarea, prompt options, quick actions, and send behavior are disabled for that URL session. The Cancel control remains enabled only while `canCancelTurn` is true; after a cancel request posts, the control still says `Cancel` but is disabled until the bridge sends a terminal event or a reconciliation error. Pressing Escape while the side panel document is focused calls `onCancelTurn()` only when `canCancelTurn` is true, including when focus is on body or non-composer UI. Enter while running must not call `onCaptureAndSend()`.

When cancellation completes through `assistant.cancelled`, partial assistant Markdown remains visible and the transcript includes the existing `Assistant turn cancelled` status card.

## What We're NOT Doing

- No protocol version bump. `session.cancel`, `assistant.cancelled`, `turn_in_flight`, and `no_in_flight_turn` already exist.
- No bridge lifecycle rewrite. Bridge cancellation is already owned by `BridgeSessionManager`.
- No provider SDK-specific cancellation logic. Issue 013 owns Codex adapter mapping.
- No queued multi-prompt UX. Same-session duplicate sends stay blocked for V1.
- No cancel affordance for prompts that are only waiting for `session.started`; there is no provider turn to abort yet.
- No New Chat behavior changes beyond preserving existing reset handling.

## Implementation Approach

Add the running/cancel state at the extension bridge/session boundary first, then route it through `UrlSessionStore` and `SidePanelController`, then render it in `SidePanelView`.

Keep React as a view. The view should render `Cancel` and forward `onCancelTurn`; it should not construct protocol messages or infer Native Messaging sequencing. The coordinator owns whether cancellation can be posted. The store owns which URL session receives the command. The controller owns whether capture, quick actions, and send commands are accepted while the active session is busy.

## Performance Considerations

Cancellation should be O(1) per active session. The main performance risk is starting capture work while a turn is already running. Phase 2 explicitly blocks `captureAndSend()` before `captureActivePageDocument()` to avoid expensive tab reads and page-context building during a duplicate send.

## Migration Notes

No persisted data migration is needed. URL sessions are in-memory. The protocol already supports cancellation in version 2.
