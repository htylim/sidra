# Enforce Context Size Limits Implementation Plan

## Overview

Implement issue 005: settings-backed readable content size limits in the extension, clear metadata-only behavior when readable text is too large, and a separate bridge-side hard payload ceiling. The goal is to prevent silent truncation while keeping capture, settings, protocol, transcript, and bridge payload policy in their existing ownership boundaries.

## Current State Analysis

Issue 005 requires a readable content limit with a V1 default, metadata-only sends when content is too large, visible `Content too large` state, bridge hard-limit rejection, and tests for under-limit, over-limit, storage defaults, live settings, and hard-limit behavior: `docs/issues/issue-005.md:13`, `docs/issues/issue-005.md:17`, `docs/issues/issue-005.md:18`, `docs/issues/issue-005.md:19`, `docs/issues/issue-005.md:20`, `docs/issues/issue-005.md:21`.

The PRD says Sidra must not silently truncate content, readable content limits live in extension settings, oversized readable content sends metadata only, and the transcript marker should read like `Page metadata attached; content too large`: `docs/prd-v1.md:281`, `docs/prd-v1.md:283`, `docs/prd-v1.md:285`, `docs/prd-v1.md:288`, `docs/prd-v1.md:289`. It also says the bridge enforces payload limits and tests should cover context size handling plus settings defaults/storage: `docs/prd-v1.md:526`, `docs/prd-v1.md:582`, `docs/prd-v1.md:583`.

Architecture assigns the relevant ownership already:

- `packages/protocol` owns message shape and Native Messaging runtime validation: `docs/ARCHITECTURE.md:18`, `docs/ARCHITECTURE.md:100`.
- `CaptureService` owns active-tab capture, extraction, size decisions, and page-context construction: `docs/ARCHITECTURE.md:92`.
- `SettingsStore` owns persisted extension settings and live settings updates: `docs/ARCHITECTURE.md:93`.
- `NativeMessagingTransport` owns frame parsing/writing and process IO; bridge-side provider lifecycle remains behind `BridgeSessionManager`: `docs/ARCHITECTURE.md:96`, `docs/ARCHITECTURE.md:98`.

Current code has the issue-004 capture path implemented:

- `PageContext` supports `readable` and `metadata_only`, but `metadata_only.reason` only allows `no_usable_text`: `packages/protocol/src/index.ts:16`, `packages/protocol/src/index.ts:27`, `packages/protocol/src/index.ts:182`, `packages/protocol/src/index.ts:188`.
- `CaptureService` extracts Readability text, falls back to `bodyInnerText`, and sends `metadata_only` only when no usable text exists. It has `MIN_READABLE_TEXT_LENGTH`, but no max-size policy: `apps/extension/src/capture-service.ts:9`, `apps/extension/src/capture-service.ts:108`, `apps/extension/src/capture-service.ts:123`, `apps/extension/src/capture-service.ts:141`.
- `UrlSessionStore` turns metadata-only context into the generic `Metadata attached` context state: `apps/extension/src/url-session-store.ts:6`, `apps/extension/src/url-session-store.ts:133`, `apps/extension/src/url-session-store.ts:240`.
- `BridgeSessionCoordinator` chooses transcript context markers from `PageContext`, but metadata-only always renders `Page metadata attached`: `apps/extension/src/bridge/session-coordinator.ts:396`, `apps/extension/src/bridge/session-coordinator.ts:399`.
- `SidePanelView` renders page-card status from `snapshot.activeSession.contextState.label`, so the view should not own size-limit policy: `apps/extension/src/side-panel-view.tsx:25`, `apps/extension/src/side-panel-view.tsx:91`.

## Desired End State

When a user triggers `Capture + Send`, Sidra extracts readable text as it does today, selects the first usable candidate by the existing rules, and checks that selected candidate against the current readable content limit. If the candidate is within the limit, Sidra sends the readable context. If it exceeds the limit, Sidra sends metadata only with reason `content_too_large`; it sends no partial readable text.

The visible page card shows `Content too large` for the URL session. The transcript shows `Page metadata attached; content too large` above the user prompt. Raw page text never appears in the transcript.

Extension settings include `readableContentLimitCharacters` with these V1 constants:

- `DEFAULT_READABLE_CONTENT_LIMIT_CHARACTERS = 120_000`
- `MIN_READABLE_CONTENT_LIMIT_CHARACTERS = 1_000`
- `MAX_READABLE_CONTENT_LIMIT_CHARACTERS = 500_000`

These are character counts over the normalized selected readable text, matching the existing `PageContext.textLength` contract. Settings load from Chrome storage, fall back to defaults when absent or invalid, and apply live to later capture attempts without reloading the side panel.

The bridge enforces a separate hard serialized payload ceiling:

- `BRIDGE_HARD_PAYLOAD_BYTE_LIMIT = 1_000_000`

This is a UTF-8 byte count over the inbound Native Messaging payload or equivalent serialized direct bridge input. Oversized payloads emit `bridge.error` with code `payload_too_large` and a generic message such as `Payload is too large.` The error must not include prompt text or captured page content.

### Key Discoveries

- The correct extension owner for size policy is `CaptureService`, not React, because architecture names capture, extraction, size decisions, and page-context construction there: `docs/ARCHITECTURE.md:92`.
- Settings storage is an intended boundary but not implemented yet. This issue should add the boundary rather than reading `chrome.storage` from capture or React code: `docs/ARCHITECTURE.md:93`.
- The protocol must grow before extension and bridge code depend on the new metadata-only reason: `docs/ARCHITECTURE.md:100`.
- The bridge hard limit is separate from extension-side content-too-large handling. Extension content-too-large is a normal metadata-only `session.send`; bridge oversized payload rejection is a `bridge.error` with `payload_too_large`.

## What We're NOT Doing

- No truncation of readable text.
- No full DOM capture or DOM size limit. That belongs to issue 006.
- No settings/options page UI. Issue 005 adds the settings storage boundary and live behavior needed by capture. A user-facing options page is outside the current issue-005 scope and should not be added here.
- No quick action settings. That belongs to issue 007.
- No token estimation or provider-specific context budgeting.
- No passive capture, background recapture, or preview of captured content.
- No Codex provider integration changes.
- No prompt or page-content logging.
- No architecture change that moves capture policy into React or raw Chrome API calls into the view.

## Implementation Approach

Use a vertical TDD sequence:

1. Extend the protocol and UI state model so `content_too_large` is representable and renders correctly without raw content.
2. Add `SettingsStore` as a dedicated extension settings boundary with defaults, validation, and live updates.
3. Inject settings into `CaptureService` and enforce the readable content limit at capture time.
4. Add a bridge-owned payload limit guard before provider code sees oversized messages.
5. Update manual E2E docs and run full validation.

The implementation should keep tests at behavior-owning boundaries. Do not add tests that assert private helper calls.

- Bridge hard-limit checks should use `Buffer.byteLength(serialized, "utf8")` for direct bridge input and the Native Messaging frame length for raw framed input.
- Oversized Native Messaging frames should be discarded without parsing JSON.
